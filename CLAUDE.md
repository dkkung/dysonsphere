## Style conventions

- Use `-` instead of em dashes (`—`) in comments, config files, and generated text.
- Maximum line length is 120 characters (`[tool.ruff] line-length = 120`). E501 is enforced across all Python files including tests and scripts. The formatter handles code automatically; comments and strings must be wrapped manually when over the limit.

## Development workflow

The loop for feature work and non-trivial changes (adapted from the process used across the v1.x cycle):

**1. Discuss before building.** For any non-trivial feature or design decision, discuss the approach first - lay out the options, their tradeoffs, and a recommendation, and get agreement before writing code. When a choice is genuinely the user's to make, ask, but recommend a default rather than only enumerating. **Make no breaking changes without consulting the user first.**

**2. Build within scope.** Implement the agreed design and keep changes within the target function/module; do not edit unrelated files without asking. Verify behavior empirically as you go (a quick script or REPL check), and trace every failure mode rather than stopping at the first plausible fix. Where correctness is subtle (something that could silently regress), add a safety-net test that would catch a future mistake. Iterate (build, run, fix) until it is right.

**3. Checkpoint - when the code for a coherent unit of work is done:**

First, **finish the Python side** - everything the checks will cover (docstrings included, since `ruff` lints their line length):
1. Update the docstrings in the files you changed.
2. Update existing tests that touch the changed code.
3. Add new tests as needed (every new public function gets tests).

Then **verify**, iterating (fix, rerun) until green:
- `uv run ruff check dysonsphere/ tests/ scripts/`
- `uv run ruff format dysonsphere/ tests/ scripts/`
- `uv run ty check dysonsphere/ tests/ scripts/`
- `uv run pytest tests/` - all pass, zero skips/failures.

Then **document** (prose the checks do not touch):
4. Add a `CHANGELOG.md` entry under `[Unreleased]` for every user-visible change (features, changes, fixes) - entries land WITH the work, in the same PR; releases only finalize + date the section.
5. Update `CLAUDE.md` if architecture, conventions, or design points changed.
6. Update memories if there is non-obvious context worth persisting.

Finally, **commit**.

**4. Commit conventions.** One-line messages with a header prefix - `add:`, `change:`, `fix:`, `remove:`, `refactor:`. No co-author / "Generated with" trailer. Commit to the working branch, never directly to `main`; branch first if needed.

**5. Pull request.** When a coherent block (a feature or working set) is done, open a PR into `main`: a crisp, outcome-focused title (no `vX.Y` prefix) and a real description (a short summary plus grouped bullets of what changed; omit a "tests and docs" section, those are implicit). No tag, no em dashes. Show the user the draft before creating it.

See `## Style conventions` for the `-`-not-em-dashes rule, the 120-character limit, and camelCase public params; see `### Release` for the release steps (including the step-0 deprecation sweep on major bumps).

## Architecture

`dysonsphere` is an [Altair](https://altair-viz.github.io/) theme and chart utility library. It wraps Vega-Lite's theme system to provide perceptually uniform palettes and publication-ready chart defaults.

### Module layout

- **`dysonsphere/palettes.py`** — All palette data as plain `dict[str, list[str]]` called `colors`. Each value is a list of hex strings (12 stops for sequential, 13 for diverging). No color math runs at import time; hex values are precomputed literals. The `palette()` helper slices/samples from this dict. **Ordering convention (v3.1.1):** the `colors` dict is grouped by **dataType → package → tier**, alphabetized case-insensitively within each group. dataType order is **sequential → diverging → qualitative** (classified by stop count: 12 → sequential, 13 → diverging, else qualitative — same rule `website/scripts/gen_palettes.py` uses); package order is **dysonsphere → `mpl_` → `cmocean_`**; tier order (dysonsphere names only) is **base → `…2` → `…3`**. Exceptions: a handful of external ramps are conceptually diverging but were resampled to 12 stops, so they're hand-placed in the diverging section (mpl: BrBG, bwr, coolwarm, PiYG, PRGn, PuOr, RdBu, RdGy, RdYlBu, RdYlGn, seismic, Spectral; cmocean: balance, curl, delta, diff, tarn, topo). Cyclic maps (mpl twilight/hsv) stay sequential. A new palette goes in its group's alphabetical slot; `colors["categorical"]` stays appended after the dict literal. **`categorical(members=1)`** returns the qualitative palette built from four base hues in the order **blue, pink, yellow, green** (`_CATEGORICAL_HUES = ("blues","pinks","yellows","greens")`) — every color is a slice of an existing base palette (`colors[hue][stop]`, stops from `(1,4,7,10)`), nothing de novo, so it is NOT a hardcoded list (retuning a base hue regenerates it; slicing is not "color math", so it respects the no-math-at-import rule). `members=1` (default) → the **flat** categorical palette, tier-major (`[colors[h][s] for s in (1,4,7) for h in HUES]`, 12 colors) so adjacent categories differ in hue — for **unrelated** groups; `members=2..10` → a **grouped** palette, hue-major (`4*members` colors) so each consecutive block of `members` categories is one hue climbing in lightness — for **paired** data (`A1`/`A2` …; sort members adjacent, pass as the color scale range). **Stops:** `members<=4` keep the exact classic tier stops `(1,4,7,10)[:members]` (byte-identical to prior versions, consistent with the flat palette's tiers); `5..10` spread evenly across the usable ramp `[1,10]` via `round(1 + 9*i/(members-1))` (0 is near-background, 11 near-black), shrinking within-hue contrast per extra member (documented; sequential per-group slices recommended for ordinal series). Raises `ValueError` below 1 or when rounded stops collide (above 10). `categorical(1)` is stored as `colors["categorical"]` (so `config.range.category` can reference it and it appears in swatch exports). Green/yellow are ordered non-adjacent (deliberate — they're the one CVD-close pair); the set is deuteranopia-clean. `export_swatches(directory=None, palettes=None, name="dysonsphere")` writes two files to `directory` (default: cwd): (1) `import_{name}_palettes_to_illustrator.jsx` — an Illustrator ExtendScript (run via File > Scripts > Other Script...) that adds the selected palettes to the active document's Swatches panel as named groups; (2) `{name}.ase` — an ASE (Adobe Swatch Exchange) binary file containing the selected palettes as named groups, automatically copied to the Illustrator User Defined Swatches folder via `_find_illustrator_swatches()` (scans macOS `~/Library/Application Support/Adobe/Adobe Illustrator */{locale}/Swatches/` and Windows `%APPDATA%/Adobe/...`); falls back to printing a copy instruction. After restarting Illustrator, `{name}.ase` appears under Open Swatch Library > User Defined > {name}. `palettes=None` (default) exports every palette in `colors`; a list selects a subset (keys of `colors`; unknown names or an empty list raise `ValueError`). `name` (default `"dysonsphere"`, preserving v1.0 filenames) renames both files and the library. `_write_ase(palette_colors, path)` generates the binary: `ASEF` header + block count + group-start/color/group-end blocks with UTF-16BE names and big-endian float RGB values. **Perceptual quality invariants are ENFORCED by `TestPaletteQuality`** (`tests/test_palettes.py`, self-contained Oklab + Machado-CVD helpers): every NATIVE palette (mpl_/cmocean_ exempt; `categorical` exempt from the sequential checks — 12 stops but qualitative) must have, sequential: monotonic Oklab L, adjacent-ΔE ratio ≤ 1.5 (empirical worst: bluelagoon 1.29), and monotonic L under Machado severity-1.0 deuteranopia AND protanopia; diverging: a V-shape (stop 6 lightest, both arms L-monotonic). A hand-edit that breaks a ramp fails CI — the bounds have headroom, so a failure means a real regression.

- **`dysonsphere/theme.py`** — `theme()` registers a custom Altair theme via `alt.themes.register` and `alt.themes.enable`. Calling `theme()` replaces the active theme globally for all subsequent Altair charts in the session.

- **`dysonsphere/export.py`** — chart → files. `save()` exports a chart to PNG, SVG (with flattened `<g>` wrappers), Vega-Lite JSON, and self-contained interactive HTML (`format="html"`) using `vl-convert`. Accepts `_AltairChart | Callable[[], _AltairChart]` where `_AltairChart = alt.Chart | alt.LayerChart | alt.FacetChart | alt.VConcatChart | alt.HConcatChart | alt.ConcatChart` (the type alias lives here). `save()` is a lean orchestrator: it drains the stats records, calls `metadata._build_block()` to assemble the `dysonsphere` block, renders each light/dark SVG/PNG, runs the SVG fixers, then calls `metadata._svg_inserts()` / `metadata._inject_png_block()` to embed. Also holds **`load()`** (rebuild the Altair object, or the raw spec dict via `raw=True`, from the Vega-Lite JSON — the inverse of `save()`, JSON-only, re-applies the baked theme via `ds.theme()`) and **`show(chart)`** (render through the full fixer pipeline and return an `IPython.display.SVG` for an accurate notebook preview — the same corrected SVG `save()` writes, but returned not written; IPython lazy-imported, raises a clear `ImportError` if absent). The render+fixer core is factored into **`_render_fixed_svg(base_obj, svg_path)`** (resolves the spec via `to_dict()`, renders via `vlc.vegalite_to_svg(spec)` — the same engine/spec `base_obj.save()` used, verified byte-identical — then **parses the SVG once**, passes the shared ElementTree root through every post-processor, and serializes once at the end - each fixer takes `root: ET.Element` and mutates it in place; none touch the file), shared by `save()` and `show()` so the pipeline stays identical. SVG namespaces are registered once at module level (`_SVG_NS`). **Tick/grid POSITIONS need no post-processing** — the theme renders with `tickRound: false` / `axisBand.tickOffset: 0` (see the exact-tick-positions design point), so ticks land exactly on their marks at render time, on every axis type and in every panel. SVG post-processors live here: `_align_grid_to_content()` (seats BOTH grid directions onto the plot content, off the detached axes - translate only, spans unchanged; on an open plot each axis is drawn `axis_offset` px away from the plot, and Vega renders each grid line inside its axis group so it inherits that offset: the vertical (x-axis) grid lines render shifted DOWN (top short of the highest tick, bottom overshooting onto the x-axis) and the horizontal (y-axis) grid lines shifted LEFT (touching the y-axis, short of the right edge); the fixer translates each back - vertical up, horizontal right - so both span the content and float symmetrically off both detached axes; skipped when the resolved offset is `0`, e.g. `closed`, so closed plots are untouched), `_fix_superscript_labels()` (corrects misaligned Unicode superscript digits in scientific/power notation labels by replacing exponents with `<tspan dy>` elements), `_simplify_svg()` (flattens redundant `<g>` wrappers for Illustrator compatibility), `_flip_ticks_inward()` (negates axis-tick line `x2`/`y2` so ticks point into the plot when `theme(inwardTicks=True)`), and `_italicize_stat_symbols()` (wraps Latin statistical symbols in italic `<tspan>`s — see its design point). Order in `_render_fixed_svg`: `_align_grid_to_content` → `_flip_ticks_inward` (only when `inwardTicks`) → `_layer_axes_to_front` → `_simplify_svg` → `_fix_superscript_labels` → `_italicize_stat_symbols` (MUST follow the superscript fixer, which only scans element `.text` — italicizing first would move label text into tspan tails it cannot see). **These SVG post-processors run in `save()` and `show()` (both via `_render_fixed_svg`), NOT in Altair's own inline renderer** — so a bare `chart` display in a notebook is approximate (superscript labels not typeset, the axisOffset grid gap present, `inwardTicks` still outward; tick positions ARE exact everywhere, since that fix lives in the theme config); use `ds.save()` (files) or `ds.show()` (inline `IPython.display.SVG`) for the accurate output. **The `html` format** (`vlc.vegalite_to_html(spec, bundle=True)`, written in the per-background loop next to the JSON block) is the *interactive* tier: self-contained (Vega JS bundled, works offline), themed, metadata-embedded (usermeta merged like the JSON), and exact-ticked (the theme config travels with the spec), but rendered live by the browser's Vega so it gets NONE of the SVG fixers. **`inwardTicks` is deliberately NOT applied to HTML:** the negative-`tickSize` trick that draws ticks inward works in vl-convert's Vega (so a vl-convert PNG "verification" falsely passed), but the browser bundles a *different* Vega build that mangles axis-label spacing with a negative `tickSize` (verified broken in a real browser) — so inward ticks stay an SVG/PNG-only feature. Depends one-way on `metadata` (`export → metadata`, never the reverse). Tests: `tests/test_export.py`.

- **`dysonsphere/metadata.py`** — the embedded `dysonsphere` block: schema, write, and read (split out of `export.py` because the metadata code had grown large and cohesive). **Write:** `_inject_png_metadata()` (generic PNG `iTXt` chunk writer), `_render_provenance()` (the "Generated by …" sentence), `_build_provenance()` (collect user/script/chart/timestamp/versions), `_call_expression()` (best-effort call-site source of `save()`'s `chart` argument - see the provenance `chart` design point), **`_build_block(records, *, embed_report)`** (assemble `{provenance, statistics, theme, report}` → returns `(usermeta, usermeta_json, report_sections)`; `usermeta_json` excludes `report` so the SVG/PNG structured blob stays report-free), `_svg_inserts()` (build the `<metadata>`/`<desc>` injection string), `_inject_png_block()` (inject description + structured + per-section report chunks, report loop `reversed()` since injection prepends). **Read:** `_iter_png_itxt()` / `_read_png_text()` (parse PNG `iTXt`), `_read_dysonsphere_block()` (unify the block from any of the 3 formats, merging the per-section `dysonsphere-report-<section>` channels back), and the public **`read(path, what="report"|"statistics"|"metadata", save=)`**. Imports `statistics`/`theme` lazily; does NOT import `export` (one-way dependency). Tests: `tests/test_metadata.py`.

- **`dysonsphere/labels.py`** — display-label helpers. **`label_expr(mapping)`** builds a Vega `labelExpr` mapping raw data values to display labels (`{"metadata_group1": "group 1"}`), usable in `alt.Axis`/`alt.Legend`/`alt.Header`. Emits a **ternary chain**, NOT the `{...}[datum.value] || datum.value` object-lookup idiom (whose `||` fallback silently misfires for falsy labels like `""`); single-quote/backslash escaping via `_js_literal` (bool checked before int — bool is an int subclass); numeric keys compare as numbers; a `list[str]` value emits a Vega array literal → **multi-line label**; unmapped values fall back to `datum.value`. Presentation-only by design: data, exported JSON, checksums, and statistics records keep raw values. Consumed by `_MarkScaffold.x_axis()` (the marks' `labelMap=`) and documented for direct use. Tests: `tests/test_labels.py`.

- **`dysonsphere/marks.py`** — Composite mark constructors: `mark_violin()` (KDE + embedded boxplot) and `mark_strip()` (jittered/beeswarm scatter + a centre statistic: with `errorbars=True` (default) a MEAN tick between mean±error bars — the tick draws the same statistic the bars are computed from, so it always sits centred between the caps (through v3.6.0 it was the boxplot MEDIAN tick, which drifted off-centre on skewed data; fixed 2026-07-11); the tick is a bare `mark_tick` on the errorbar summary frame - ALL styling inherits `config.tick` (see its design point), which also resolves darkmode at render time (no callable needed for correct tick colour across `save()` backgrounds, unlike the call-time `_opt("darkmode")` reads); with `errorbars=False` the hidden-boxplot median tick, styled by `markMedianFill`, remains); shared chrome lives in the `_MarkScaffold` dataclass (see its design point). Both return Altair objects for composition with `+`. Safety net: `test_errorbars_center_tick_is_mean` in `tests/test_marks.py`.

- **`dysonsphere/table.py`** — `mark_table(df, columns=None, ...)` renders a DataFrame as a publication-styled table, returning an `alt.LayerChart` that composes with `+`/concat. **Everything is laid out in PIXEL space** (`alt.value` x/x2/y/y2, one shared ordinal row-band y-scale on a `transform_window` `__rowidx`, no x-scale) so it drops into concat without scale merging — like `add_shade`'s pixel mode / `multilabel`. **The df is NEVER mutated and stays fully recoverable:** every per-row mark (stripes, value-color rects, cell text) rides on `alt.Chart(df)` (Altair dedupes the shared frame to ONE inlined dataset), with the row index from `transform_window(__rowidx="row_number()")` and formatted labels / contrast colours from `transform_calculate` — **transform fields are computed at render, never inlined into `data.values`**, so `read(what="data")` and the provenance `dataChecksum` return the exact frame passed in (safety net: the `mark_table` entry in `test_read_data_filters_internal_sidecars`). Only the fixed chrome (stroke rules, header text) rides on tagged `_internal_data` sidecars. **`strokes`** is a composable set (`"outer"`/`"header"`/`"rows"`/`"cols"`/`"all"`=rows+cols; a str is accepted; default `("outer","header")`). **Striping** shades alternating rows with **one `mark_rect` per CELL** (a `len(cols) × nStripes` grid of parity-filtered layers, each at one column's x-span) - never a full-width per-row rect - so every cell background is an independent `<rect>`, individually editable in Illustrator (like the already-per-cell `cellColor` rects). **Seam fix (both directions):** abutting per-cell rects leave a sub-pixel gap that shows the page through at the browser's chart zoom (invisible at print DPI, so a PNG looks clean - the same seam the gallery heatmaps avoid). Fixed by overhanging each cell into the next by `_seam` (0.5px) on BOTH axes: `_cell_x2(i)` = `min(right+_seam, total_w)` (column edge), and the rows are positioned in PIXELS not a band scale - `_df_base` adds `transform_calculate` `__ytop`/`__ybot`/`__ymid` (`__ybot = min(header_h + rowidx*rowHeight + _seam, total_h)` overhangs the next row, last row clamped), with an identity linear `y_scale` (`domain==range==[0,total_h]`, `nice/zero` off) via `_y(field)` so rects span `[__ytop,__ybot]` and text sits at `__ymid`. **Why pixel rows, not a band scale:** Altair rejects a negative band `paddingInner` (the tidy way to overlap bands), so the band scale was replaced with the identity-mapped pixel scale, which every df-driven layer shares (header/strokes use `alt.value` pixels directly). The overhang is invisible between same-colour stripe cells; a value-coloured cell bleeds ≤`_seam` px into its neighbours, hidden under any cell stroke. Constant fill, stroke pinned off to dodge the `config.rect` leak; light mode uses the palette's lightest `nStripes` stops, dark mode the darkest (`_opt("darkmode")` at build → callable needed for `save()` across backgrounds). **`columnFormat`** (`{col: spec}`) reuses the notation vocabulary: `"scientific"`/`"power"` render Unicode superscripts via a `transform_calculate` expression (`_calc_expr`, reusing `nonlinear._SUP`) that the SVG fixers then typeset; `"e"`/`"si"`/printf-specs go through native `alt.Text(format=)`; unlisted numeric columns default to `sigFigs` sig-figs; strings render verbatim. Display strings are ALSO computed in Python (`_fmt_scientific`/`_fmt_power`/`_fmt_si` + `format()`) for the content-based column-width estimate (Vega can't measure text at build), so the superscript notations format twice (Python for width, Vega for render) — a rare mantissa-carry mismatch is invisible at estimate precision. **`cellColor`** (`{col: palette}`) shades a numeric column by value (a `mark_rect` color scale on the Q field; 13-stop diverging palettes get a symmetric `[-m, m]` domain, else `[min,max]`); text auto-contrasts via `_contrast_expr` (a `transform_calculate` ternary over `datum[col]` → `'black'`/`'white'` at WCAG-luminance crossings, `_rel_luminance`), encoded `alt.Color(scale=None)`. Multiple value-color scales + the contrast color scales are isolated with `.resolve_scale(color="independent")`. **Text/header colours** (all darkmode-aware at build, so a `save()` across backgrounds needs a callable): **`textColor`** (`str` all-cells / `{col: color}` dict / `None` inherit) resolved by `_text_color(col)` → `("fixed"|"contrast"|"inherit")` with the precedence **dict-entry (wins even on a `cellColor` column) → `cellColor` auto-contrast → global str → inherit** (a global str deliberately does NOT override a heatmap column's contrast; a per-column dict entry does); **`headerColor`** (`str`/`None`) sets header text, else auto-contrasts against `headerFill` when set, else inherits; **`headerFill`** (`str | bool`: `False` none / `True` a darkmode grey band `greys[3]` light / `greys[9]` dark / a str) draws a full-width `mark_rect` band behind the header (bottom layer). Alignment defaults **all columns left** (`_col_align`; override via `align` str or `{col: side}` dict). **`fontStyle`** (`str` all-cells / `{col: style}` dict / `None` inherit; `_font_style(col)`) sets the body cell font style - e.g. `{"gene": "italic"}` for italic gene names (the header p in "adj. p-value" italicises via the shared stat-italicizer, not this). Column widths estimate `max(header,cells)·fontSize·0.6 + 2·cellPadding` (override via `columnWidths` list/dict); the table sets its own `width`/`height` (overriding the 100×100 default — the documented exception to the test-at-default rule). `mark_table` is in `__all__` + `__init__.__all__` + `test_package.py`'s `_MODULE_NAMES`. Build script: `scripts/build/build_table_example.py` → `docs/table_example.png`. Tests: `tests/test_table.py`.

- **`dysonsphere/annotations.py`** — Composable non-statistical annotations (split from the former `layers.py` in v3.0): `add_rule()` (horizontal / vertical reference lines, `axis="y"` / `"x"`), `add_text()` (text annotations at data or pixel coordinates, with 9-position presets), `add_shade()` (background `mark_rect` shading layer), and `add_labels()` (auto-placed point labels with connectors; the pixel placement engine lives in `_placement.py`). Shared private helpers: `_rule_mark_kwargs`, `_rule_label_geometry`, `_datum_base`, `_datum_ref_layers`, `_is_alt_value`, `_text_datum_layers`, `_TEXT_PRESETS`. These return `alt.Chart` or `alt.LayerChart` objects for composition with `+`.

- **`dysonsphere/inference.py`** — Statistical inference annotations (split from the former `layers.py` in v3.0; named for inferential statistics - hypothesis tests + correlation estimation - the engine↔wrapper pair with `statistics.py`): `add_comparisons()` (pairwise p-value brackets with auto-stacking, OR omnibus corner label + post-hoc brackets; internal helpers `_pvalue_layer()`, `_omnibus_label()`, `_bracket_pvalues()`, `_format_pvalue()`, `_format_asterisks()`, `_superscript()`) and `add_correlation()` (scatter correlation coefficient + OLS fit line for Pearson; helper `_correlation_label()`). Delegates pure statistical computation to `statistics.py`; imports `add_text` from `annotations.py` (one-way). **`add_comparisons` was named `add_pvalue` through v1.0 (renamed in v1.1); the deprecated `add_pvalue` alias was removed in v2.0.** Grep `DEPRECATED` before any major bump.

- **`dysonsphere/_placement.py`** — Pure pixel-space label placement (no Altair import), backing `annotations.add_labels`: `_repel_labels` (nearest-clear-spot + 2-opt; see the add_labels design point) and `_sample_spread` (deterministic farthest-point sampling for `labels=n`). The engine↔wrapper pattern mirrors `statistics.py` ↔ `inference.py`. Tests: `tests/test_placement.py`.

- **`dysonsphere/multilabel.py`** — Condition table compositor: `add_multilabel()` attaches an annotation table below a chart by stripping its x-axis labels and vconcat'ing with a `_multilabel_layer()` annotation. Accepts `alt.Chart | alt.LayerChart` (e.g. a strip+boxplot layer). `_multilabel_layer()` is the private implementation. Returns `alt.VConcatChart`. Split from the annotation family (the former `layers.py`) because the multilabel system is a chart compositor (not a composable layer), and has grown sophisticated enough to justify its own module.

- **`dysonsphere/nonlinear.py`** — Minor tick constructors for non-linear axes: `add_log_ticks()` (unlabeled minor ticks for log-scale axes, base 10 or arbitrary integer base) and `add_pow_ticks()` (unlabeled minor ticks for power/sqrt-scale axes, arbitrary exponent). Private helpers: `_log_minor_layer()`, `_derive_exp()`, and `_infer_field()` (recovers the `field` from the chart's x/y encoding when omitted) for `add_log_ticks()`; `_pow_minor_layer()` for `add_pow_ticks()`. Both public functions use the invisible `mark_point(opacity=0)` + independent-axis trick.

- **`dysonsphere/statistics.py`** — Pure statistical computation (no Altair import), backing `add_comparisons()` in `inference.py`. Holds: omnibus runners `_run_omnibus()` (ANOVA `f_oneway`, Kruskal-Wallis, Friedman, Alexander-Govern) returning an `_OmnibusResult` dataclass; hand-rolled post-hoc matrices `_dunn_matrix()`, `_games_howell_matrix()`, `_nemenyi_matrix()` (scipy primitives — `rankdata`/`norm`/`studentized_range` — validated against scikit-posthocs/pingouin) plus `_tukey_matrix()`, dispatched + corrected by `_post_hoc_matrix()`; effect sizes `_eta_squared()`/`_epsilon_squared()`/`_kendalls_w()` (omnibus) and `_pair_effect()` (Cohen's d / rank-biserial, per pair); `_adjust()` (bonferroni/holm); descriptives `_describe()`/`_describe_all()` (`sd` is `None` for n≤1, not NaN, so the record is JSON-safe); the structured-record builder `_make_record()` and its text renderer `_render_report()`; and the report registry `_REPORTS` (a `dict[str, dict]` keyed by content hash) with `_register_report()` (stores + returns a unique marker `name` `__dysonsphere_<hash>_<counter>`), `_marker_hash()` (extract the hash from a marker name), `_select_reports(hashes)` (records for the given hashes), and public `clear_stats()`. Also **correlation** (backing `add_correlation`): `_run_correlation(kind, x, y)` (`pearson` via `linregress` → coefficient/r²/p/slope/intercept; `spearman`/`kendall` via `spearmanr`/`kendalltau` → coefficient/p only, no line), `_make_correlation_record()`, and `_render_correlation()` (a `_render_report` branch). Constants: `_OMNIBUS_TESTS`, `_OMNIBUS_NAMES`, `_POSTHOC_DEFAULTS`, `_PAIRWISE_TESTS`, `_PARAMETRIC_POSTHOC`, `_EFFECT_NAMES` (symbol→machine name, e.g. `η²`→`eta_squared`), `_CORRELATION_METHODS` (method→(display, symbol, machine_name)). (Earlier this module held `add_comparisons` itself; that role inverted in v1.1 — the annotation wrapper moved out, to the module now called `inference.py`.)

- **`dysonsphere/transforms.py`** — Pure data transforms that add columns to a Polars DataFrame: `add_jitter()` (Gaussian x-offsets) and `add_beeswarm()` (analytic collision-avoiding x-offsets). Output columns are then passed to Altair's `xOffset` encoding. `add_jitter()` `spread` defaults to `min(chartWidth, chartHeight) / 50` (2.0 at 100×100), matching the `markSize` scaling pattern.

- **`dysonsphere/utils.py`** — Shared utilities. Contains **`band_geometry(n, span=None, *, scale, bandPadding=None)`** — the single source of truth for band-axis pixel math, returning a `BandGeometry` named tuple `(step, centers, starts, ends)`. Three `scale` variants: `"offset"` (default; paddingInner=0, paddingOuter=bp — what xOffset/`mark_circle`/`add_shade` rects see; `step = span/(n+2bp)`), `"band"` (paddingInner=paddingOuter=bp — what `mark_boxplot`/`mark_violin` see; `step = span/(n+bp)`), and `"point"` (`step = span/n`). `span`/`bandPadding` default to the theme's `chartWidth`/`bandPadding` (pass `chartHeight` for a y-axis). Consumers: `mark_violin` centres + `mark_strip` step (marks.py), `add_shade` all three modes (annotations.py) + the `add_comparisons` bracket midpoint (inference.py), `add_multilabel` spans (multilabel.py). Do NOT hand-roll `chartWidth / (n + …)` band math anywhere — extend/consume this helper. Tests: `TestBandGeometry` (includes a rendered-boxplot equivalence check). Also contains `ensure_polars(df)`, which accepts a `polars.DataFrame` or `pandas.DataFrame` and always returns a `polars.DataFrame`. Called at the entry point of every public DataFrame-accepting function (`mark_violin`, `mark_strip`, `add_comparisons`, `add_jitter`, `add_beeswarm`). Pandas is detected via `type(df).__module__.startswith("pandas")` to avoid a hard import; conversion uses `pl.from_pandas()`, which requires `pyarrow` for object-dtype (string) columns. Also contains `count_n(df, xCol, categories)`, which returns a `list[int]` of per-category row counts in `categories` order; calls `ensure_polars` internally so it accepts both DataFrame types. Also `_internal_data(data)` + `_INTERNAL_COL = "__dysonsphere__"` — the **internal-data sentinel** (see the discipline point below). Also **`_hash_rows(rows)`** (order-independent `sha256:` of a list of row dicts — the multiset-of-per-row-digests core shared with the provenance `metadata._data_checksum`) and **`frame_checksum(df)`** (`ensure_polars(df).to_dicts()` → `_hash_rows`), used to tag statistics records with their source-dataframe identity (see the record `dataChecksum` design point). Also **`_nice_domain(lo, hi, count=10)`** — d3's `nice()` (1/2/5/10 tick-increment outward rounding, applied twice), used by `add_labels` to pin round axis bounds instead of the raw data extent; exactness vs Vega doesn't matter because the caller forces the returned domain (self-fulfilling). Tests: `TestNiceDomain`.

**Internal-data sentinel — DISCIPLINE (v1.1):** dysonsphere's composite marks/annotations generate their own small "sidecar" datasets (bracket coords, mean/error bars, KDE curves, labels, span/shade rects, …). Altair inlines each as a separate named dataset alongside the user's dataframe, so `export.read(what="data")` must tell them apart. Every internal data source is tagged with the `utils._INTERNAL_COL` (`"__dysonsphere__"`) column via **`utils._internal_data(...)`**; `read(what="data")` filters any dataset carrying it, returning only the user's frame(s). **RULE: any code (current OR future) that builds a dysonsphere-generated data source for a chart layer MUST route it through `_internal_data(...)`** — i.e. `alt.Chart(_internal_data(rows_or_df))` (accepts a `list[dict]` → `alt.Data`, or a polars/pandas df → tagged polars df). Miss one and that sidecar leaks as a phantom "user" dataframe (→ a false multi-frame error, or the wrong frame). Do NOT tag the USER's data (e.g. `mark_strip`/`mark_violin`'s points/boxplot on `df`, or `add_log_ticks`'s invisible layer which reuses the user's `df` for the scale — tagging it would duplicate the user's data). Tagged sites (grep `_internal_data`): `add_rule`/`add_text`/`add_shade`/`add_labels` in annotations.py, `add_comparisons`/`add_correlation` in inference.py, `violin_df`+`summary` in marks.py, all of multilabel.py's annotation dfs + `_one_row`. NOT nonlinear.py (its minor ticks are axis config, not data). Note the sentinel column `"__dysonsphere__"` shares a prefix with the stats *marker* name `__dysonsphere_<hash>` — they're distinct (a data column key vs a layer `name` field); loose `"__dysonsphere_"` substring checks match both. Safety-net test: `TestSaveUsermeta.test_read_data_filters_internal_sidecars` renders every chart type and asserts exactly one user frame.

- **`dysonsphere/discovery.py`** — Optional-extension discovery (see the extension-architecture design point). Holds the entry-point group constant `_ENTRY_POINT_GROUP = "dysonsphere.extensions"`, `_extension_entry_points()` (installed EPs keyed by name via `importlib.metadata.entry_points(group=...)`), and the public **`extensions()`** (sorted installed names) / **`load_extension(name)`** (import one by name; `ImportError` naming the installed set if absent). Also the **extension-usage provenance** helpers: `_EXT_MARKER_PREFIX = "__dysonsphere_ext_"`, **`_tag_extension(chart, name)`** (tag a chart with a durable view-`name` marker so `save()` records the PRODUCING extension - reuses the stats-marker channel that survives `+`; re-exported publicly as `ext.tag_extension`), and **`_used_extensions(spec)`** (walk a spec for those markers → `{name: version}` from the installed EPs' dists - the extensions that actually produced the figure, not merely installed; skips a marker whose EP/dist is absent). `metadata._strip_markers` already removes the `__dysonsphere_`-prefixed ext markers from written output for free. NOTE the module is named `discovery.py`, NOT `extensions.py`, to avoid colliding with the exported `extensions()` function (a `from .extensions import *` would overwrite the submodule attribute with the function). Tests: `tests/test_extensions.py`.

- **`dysonsphere/ext.py`** — The **public primitive surface for extension authors** (`dysonsphere.ext`; see the extension-architecture design point). A stable, versioned re-export of the `_`-private core primitives an extension needs to build first-class composites, so extensions never reach into `utils._internal_data`/`theme._opt`/`export._AltairChart` directly (those may change; the `ext` names go through deprecation like any public API). Kept **minimal on purpose** — a primitive is promoted only when a real consumer needs it (growing a public surface is cheap/non-breaking; shrinking one is not). **Current set (exactly these):** `AltairChart` (the chart-object union, for return annotations), `opt(key)` (read an active-theme option — the only supported theme-option read outside core), `internal_data(data)` (tag a GENERATED sidecar dataset so `read(what="data")` filters it — same discipline as core's `_internal_data`: tag data you computed, never the user's frame), `tag_extension(chart, name)` (tag a chart your extension built so `save()` records its version in `provenance.environment["dysonsphere-extensions"]` — a durable marker surviving `+`, stripped from output; re-exports `discovery._tag_extension`). Exposed as the **namespaced `ds.ext`** (bound via `from . import ext` in `__init__.py`); deliberately NOT star-imported onto `ds.*`. Growing the set (`register_report`, `MarkScaffold`, `datum_base`/facet-safety, `band_geometry`) is deferred until a plot needs it (`tag_extension` was the first such promotion — the volcano's provenance self-tagging). Tests: `tests/test_ext.py` (re-export identity + a `_volcano_like` dogfood composite that round-trips through `save`/`read` to prove the surface delivers sidecar filtering).

- **`dysonsphere/__init__.py`** — Star-imports all modules; everything public is available directly on the `dysonsphere` namespace. **Every module defines `__all__`** (v3.0) so the star-imports bind exactly the intended API - ~31 names, no leaked stdlib/third-party imports (pre-v3 `ds.np`, `ds.math`, even `ds.field` were importable); `__init__.__all__` is the explicit union, guarded by `tests/test_package.py`. A new public function must be added to its module's `__all__` AND `__init__.__all__`. The package ships a `py.typed` marker (PEP 561), so annotations are part of the public contract. Also defines the package-level **`__getattr__`** (PEP 562) that lazily resolves installed extensions as attributes: `dysonsphere.biology` imports/returns the `dysonsphere-biology` extension registered under the `dysonsphere.extensions` entry-point group and caches it in `globals()`; any other missing attribute raises the usual `AttributeError` (delegates to `discovery._extension_entry_points`). `extensions()`/`load_extension()` (from discovery.py) are star-imported and part of `__all__`; `ext` is bound as the namespaced `ds.ext`, its contents deliberately OFF the top namespace.

### Key design points

**Extension architecture (Option D — separate distribution + lazy discovery):** optional domain packages (`dysonsphere-biology`, later `-physics`/`-astronomy`) are **independently released distributions**, NOT extras and NOT a namespace package. Each ships its own top-level package (`dysonsphere_biology`, importable directly) with its own heavy/optional deps (e.g. pyBigWig/pysam/bioframe), lazy-imported with a clear `ImportError` — quarantined from core's release cadence. It registers itself under the `dysonsphere.extensions` entry-point group in its `pyproject.toml` (`[project.entry-points."dysonsphere.extensions"]` → `biology = "dysonsphere_biology"`). Core stays a **regular package** (no PEP 420 namespace surgery, no break to `import dysonsphere as ds; ds.theme()`); the only core hook is the additive package `__getattr__` + `discovery.py`, giving the ergonomic `ds.biology.volcano(df)` path (resolved + cached on first access) plus `ds.extensions()` to enumerate what's installed. Rejected alternatives: a pure PEP 420 namespace (`dysonsphere.biology`) would force core code to `dysonsphere.core.*` — a large break for cosmetic gain; extras (`dysonsphere[biology]`) couple release cycles and bloat one wheel; pkgutil-style namespaces are fragile/legacy. **Public primitive surface (`dysonsphere.ext`) — done, minimal-first:** the discovery layer is complete AND a minimal `dysonsphere.ext` SDK now re-exports `AltairChart`/`opt`/`internal_data` (see the ext.py module entry). Deliberately minimal: `register_report`, `MarkScaffold`, `datum_base` (facet-safety), and `band_geometry` are **NOT** exposed yet — promote each only when a concrete extension plot needs it (adding to a public surface is non-breaking; removing is not, and several of those primitives carry still-evolving schemas/invariants easy to misuse).

**`dysonsphere-biology` package + uv workspace (in-repo dogfood):** the first extension lives in this repo as a **uv workspace member** — subdir `dysonsphere-biology/` (own `pyproject.toml`, top-level pkg `dysonsphere_biology`, entry point `biology = "dysonsphere_biology"`, depends on the local editable `dysonsphere` via `[tool.uv.sources] dysonsphere = {workspace = true}`). Root `pyproject.toml` declares `[tool.uv.workspace] members = ["dysonsphere-biology"]`, a dev-dep on `dysonsphere-biology`, and `[tool.uv.sources] dysonsphere-biology = {workspace = true}`; `uv sync` installs both editable so `ds.biology.volcano` resolves through the REAL entry point (dogfooding discovery + `ext` together, not a simulation). Splittable to its own repo at release time. **Run its tests too:** `uv run pytest tests/ dysonsphere-biology/tests/` (biology has its own `conftest.py` mirroring core's config-isolation + report-queue fixtures — otherwise a user-wide `saveBackground` bleeds in), and include `dysonsphere-biology/` in the ruff/ty commands. First chart: **`volcano(df, ...)`** (`dysonsphere_biology/volcano.py`) — log2FC vs −log10(p), classified up/down/ns (colored `pinksblues` endpoints + `greys[1]` ns, darkmode-aware), optional threshold guides (`ds.add_rule`) + gene labels via **`ds.add_labels`** (force-repel + connectors; the volcano does its own significance-ranked selection - top-N by `|log2fc|·−log10p`, all-significant, or an explicit list - and passes the chosen names as `add_labels(..., labels=[names])`, since its ranking is domain-specific rather than `add_labels`'s spatial even-spread); built entirely on public surfaces (core + `ext`) as the reference for extension authors. Example: `dysonsphere-biology/examples/volcano_example.py`.

**Axis-title clobber — FIXED in core (2026-07-05):** `add_rule`/`add_text`/`add_shade` used to null the base chart's axis title (a `title=None` field on the shared channel wins Vega-Lite's layer-merge); all three now position by `alt.datum`/`alt.value` (see the `add_rule` design point). So `volcano` just uses plain `ds.add_rule` for its threshold guides - no workaround needed. **Next:** grow `ext` as later biology plots demand; more biology plots (manhattan/MA, deepTools, genome browser).

**Palette construction:** Custom palettes are built in [Oklab](https://bottosson.github.io/posts/oklab/) for perceptual uniformity. `scripts/build/print_palettes.py` contains the four recipes (single-hue, multi-hue, diverging, chroma-scaling) and arc-length resampling logic. Running it prints new hex literals to stdout; you then paste them into `palettes.py` manually. `build_multihue` takes an optional `max_chroma` cap (per-palette overrides in `SEQ_MULTI_MAX_CHROMA`): paths crossing the sRGB blue gamut bulge (hue ~250-280) pick up a chroma spike that bunches the lightness steps under arc-length resampling; the cap flattens the spike so the L ramp stays even (greyscale-safe). Used by `nebula` (0.14) — the viridis-alternative multi-hue (deep slate-teal → indigo → periwinkle → orchid → pale rose), stored DARK-first (viridis polarity; multi-hue storage direction is per-palette: dusk/ember/moss are light-first, lagoon/shoal/nebula dark-first) — and by `cosmos` (0.14), nebula's magma: the same hue route shifted darker (L 0.22-0.80 since the 2026-07-11 floor lift; deep-teal floor) so the light end stops at a saturated pink (`#FC9BC6`) instead of near-white rose; cosmos also overrides `frac=0.95` (per-palette overrides in `SEQ_MULTI_FRAC`; precedent: oranges/yellows/neongreens) to supply the pink's chroma — and by `borealis` (0.14, frac 0.95, dark-first), the viridis analogue: rich violet floor → cobalt → azure-teal → luminous emerald (L 0.22-0.86 since the floor lift, saturated at BOTH ends, unlike the near-white-topped house ramps). Verified: adjacent-stop ΔE ratio 1.07 (viridis 1.05), L monotonic + healthy adjacent ΔE under Machado deuteranopia AND protanopia (protanopia still beats viridis) — and by `australis` (0.14, frac 0.95, dark-first), borealis's journey extended through cosmos's magenta territory: deep magenta-wine fringe → violet → cobalt → azure → teal → the same emerald curtain (L 0.22-0.87 since the floor lift, ~190° of hue rotation, the widest in the collection; adjacent ΔE ratio 1.08; CVD-monotonic with MORE capacity than viridis under both dichromacies). The nebula/cosmos/borealis/australis quartet is the "celestial family" of viridis alternatives (borealis was briefly named `aurora`, renamed pre-release when australis joined). `max_chroma` also accepts a **per-keyframe LIST** (interpolated alongside L and hue — the cividis trick: saturated ends, near-grey middle so a hue crossing passes through neutral instead of olive), used by the **metal pair `brass`/`pewter`** (deliberately NOT celestial-named; the shared metal register marks them as siblings): deep navy → near-grey petrol → metallic top (brass = vivid gold `#FDDB72`, pewter = warm platinum `#ECE5CB`), frac 0.95, caps `[0.12,0.09,0.035,0.09,0.13]` / `[0.12,0.09,0.035,0.045,0.035]`. They fill the cividis seat: the blue→gold axis survives dichromacy fully (L monotonic, adjacent ΔE min ≥ 0.06 under Machado deuteranopia and protanopia). **Floor lift (2026-07-11):** australis/borealis/cosmos/brass/pewter had near-black floors (L 0.13-0.15) whose darkest stops read as indistinguishable black; all five were re-centered to floor L 0.22 by linearly remapping the keyframe lightnesses (hue routes + chroma caps unchanged; nebula at 0.21 untouched). Capacity lands at viridis parity (ΔE arc: australis 0.797 vs viridis 0.806, still CVD-superior) - the near-black floor had been buying a capacity SURPLUS, not the standard. The pre-lift keyframes are preserved as comments in print_palettes.py (byte-identical regeneration; the shipped originals live in 3.6.x). **`eclipse` (2026-07-11, the 7th of the cycle):** the monochrome identity ramp - warm ink -> warm paper (constant hue 90, frac 0.95, chroma cap 0.012 = the website brand-chip register `#EEECE6`, deliberately NOT tan; cap 0.025 read khaki and was rejected), stored dark-first. **Deliberately EXEMPT from the 0.22 floor rule** (L 0.17-0.975): the floor lift prevents dark-HUE confusion, which a monochrome ramp cannot have - and since lightness is its only encoding channel, the stretched span is what buys exact viridis-parity capacity (dE arc 0.807 vs 0.806) with ZERO capacity loss under either Machado dichromacy (0.807/0.806, best in the collection) and the best greyscale/print fidelity possible (arc == L span for a mono ramp; its inherent trade is no hue redundancy). Named for the mono logo (black shell, pale corona); alphabetical slot between dusk and ember. **`condensates` (2026-07-11) - the imaging colormap:** a purpose-built true-black-floored map for fluorescence micrographs / astronomy, where 0 signal must render `#000000` AND dim structure must stay legible. Built by `print_palettes.build_condensates()` (alphabetical slot between candy and cosmos): a **25/75 Oklab blend** of two constructions of australis's hue journey - **V1** australis hues placed on **magma's front-loaded lightness curve** (`CONDENSATES_MAGMA_L`, magma's Oklab L baked at the 12 stops, no matplotlib dep) and **V2** australis's keyframes lightness-stretched to the full `[0, 0.97]` range (near-black -> near-white mint top `#E3FFE6`) - with stop 0 forced to pure `#000000`. **Why this design (the key finding):** true-black floor + magma-class low-signal detail + Oklab-uniformity ≤1.5 are mutually exclusive at the extreme - magma/inferno themselves score Oklab ratio 1.76/1.70 (they'd FAIL our test), because preserving dark detail *means* front-loading lightness, which is non-uniform. The naive `<name>-black` approach (prepend black to the existing journey) crushed the low range (~2x magma's dark-entry count). The lever is the **lightness ceiling**: spreading the hues over the full black->near-white range (V2) is what buys the detail; blending in 25% of magma's exact curve (V1) lifts the dim mid-tones further while keeping V2's crisp puncta-on-black look. At 12 stops the blend lands at ratio **1.28** - Oklab-uniform, L-monotonic, CVD-monotonic under deuteranopia+protanopia - so `TestPaletteQuality` checks it like any native sequential palette (no exemption). This SUPERSEDES the abandoned true-black `-black` family (a `<name>-black` for all seven cycle ramps, briefly on PR #121): true-black + detail + uniformity can't be had across arbitrary palettes, so it ships as ONE dedicated australis-derived imaging map instead. The website gallery's condensate micrograph is recolored with it.

**Theme registration:** `theme()` takes `style: str | None = None` and `**kwargs`. All built-in defaults live in `_BUILTIN_DEFAULTS`; the built-in named preset (`"notebook"`; `"presentation"` was removed in v3.0) lives in `_BUILTIN_STYLES` — both dicts in `theme.py`. Merge order (ascending priority): `_BUILTIN_DEFAULTS` → `[default]` from config file(s) → `_BUILTIN_STYLES[style]` → `[style]` from config file(s) → explicit kwargs. Built-in presets work without any config file; the config file only customises them. `create_config(directory=None, *, persist=False)` scaffolds a `dysonsphere.toml` with all built-in presets at their defaults plus an empty `[my_style]` placeholder; `persist=True` writes to the platform user config dir (`_user_config_dir()`: XDG on macOS/Linux, `%APPDATA%` on Windows) instead of cwd. `_config_paths()` returns config file paths in ascending priority order: `$XDG_CONFIG_HOME/dysonsphere/dysonsphere.toml` (user-wide) then `dysonsphere.toml` found by `_find_project_config()`, which walks up from `Path.cwd()` to the filesystem root (git-style) and returns the first match. `_load_style_overrides(style)` raises `ValueError` for unknown TOML keys, or if the style is not in `_BUILTIN_STYLES` and not found in any config file. `_load_custom_palettes()` reads `[palettes]` sections from config files and returns a `dict[str, list[str]]`; each value must be a non-empty list of strings. On each `theme()` call, `colors` is restored to `_ORIGINAL_COLORS` (snapshot at import time) then updated with custom palettes, so they reset cleanly when no config file is present. Every palette-valued key (`palette` + the five `<type>Palette` keys) is resolved in one loop: `colors[val] if isinstance(val, str) and val in colors else val` — so a name resolves to its hex list, while a raw list or a Vega scheme name (e.g. `"tableau10"`) passes through (the `isinstance(val, str)` guard avoids a `TypeError` from `list in colors`). `alt.theme.options` is set via `{**p, "tickWidth": p["axisWidth"]}` in one shot.

**`_opt()` theme-option accessor:** all theme-option reads outside theme.py go through **`theme._opt(key)`** — never `alt.theme.options.get(key, hardcoded)` (the old pattern's ~60 per-site hardcoded fallbacks could silently drift from `_BUILTIN_DEFAULTS`). `_opt` reads `alt.theme.options[key]` and falls back to a cached, fully **derived** copy of the built-in defaults (`_compute_derived(p)` — the computed-defaults block factored out of `theme()`, resolving the None/True sentinels: `markSize` 10.0, `axisOffset` 4.5, `markStrokeWidth` = axisWidth, `closed` False, …), so a helper called before any `ds.theme()` sees real values, not sentinels. Unknown keys raise `KeyError`. Direct `alt.theme.options[...]`/save-restore writes are unaffected (e.g. `save()`'s darkmode toggling). Tests: `TestOptAccessor` in `tests/test_theme.py`.

**Per-type range palettes (v1.1):** `categoryPalette`/`divergingPalette`/`heatmapPalette`/`ordinalPalette`/`rampPalette` (all `None` in `_BUILTIN_DEFAULTS`) override individual `config.range.*` schemes. Because they live in `_BUILTIN_DEFAULTS` they are automatically valid as both `theme()` kwargs AND TOML `[default]`/`[style]` keys (no change to validation/merge needed). Resolution in `_dysonsphere_theme()` uses a local `_scheme(type_key, default)` helper with precedence **`palette` (master) → `<type>Palette` → built-in default** — i.e. the global `palette`, when set from any source, overrides the per-type keys (per user decision; inverts the usual specific-beats-general). Per-key source order is the standard merge (kwarg > TOML > default), so an explicit `theme(palette=…)` is the highest thing overall. Calling `theme()` again replaces the theme entirely. **Built-in range defaults:** `category` → `colors["categorical"]` (the multi-hue qualitative palette, see palettes.py `categorical()`), `ordinal` → `colors["greys"]`, `heatmap` AND `ramp` → `colors["australis"]` (2026-07-10, was `blues`; the viridis-analogue journey, stored dark-first so low values are dark — viridis polarity), `diverging` → `colors["pinksblues"]`. **`category` is special — it is emitted as a BARE array, not `{"scheme": …}`** (`category_range = _cat if isinstance(_cat, list) else {"scheme": _cat}`, resolved just before the return dict): a nominal color scale maps a bare array **positionally** (category *i* → color *i*, which the tier-major hue cycling requires), whereas `{"scheme": [array]}` is invalid for nominal and silently drops the range (verified: bare array renders positional, scheme-wrap renders no fills). The `isinstance` guard keeps a Vega scheme *name* (a `str`, e.g. `"tableau10"`) wrapped as `{"scheme": name}` since a bare string isn't a valid range. The other four range types stay `{"scheme": …}` (sequential/interpolated is correct for them). Overriding a base hue via TOML `[palettes]` does NOT ripple into `colors["categorical"]` (derived once at import; decoupled by design) — override `categorical` directly, or use `categoryPalette`. `dashedGrid=False` controls whether axis grid lines are dashed (uses the `dashedWidth` pattern); grid lines are independent of `dashedRule` which only affects `mark_rule` marks.

**Chart title config:** The `"title"` block in `_dysonsphere_theme()` sets `"anchor": "middle"` and `"frame": "group"` explicitly. Both are required: `anchor` alone is not picked up by Vega-Lite when the title is a plain string (e.g. `.properties(title="...")`); `frame="group"` ensures centering is relative to the chart's layout group (view + axis space) rather than the rendered mark bounds. Together they make `.properties(title="...")` behave identically to `alt.TitleParams(text="...", anchor="middle", frame="group")`.

**`cornerRadius` parameter:** `cornerRadius: float | bool = False` in `_BUILTIN_DEFAULTS`. `False` → omitted (no rounding). `True` → resolved to `min(chartWidth, chartHeight) / 100` at theme-call time (1 px at the default 100×100 chart). Explicit `float` → used as-is. Applied as `config.rect.cornerRadius`, `config.boxplot.box.cornerRadius`, and `config.arc.cornerRadius` (all corners on each); and `config.bar.cornerRadiusEnd` (tip only, not `cornerRadius`). `mark_square` was considered but excluded — it renders as an SVG path symbol, not a `<rect>`, so `cornerRadius` has no effect on it. The `if opts["cornerRadius"]` guard omits the key entirely when falsy so Vega-Lite's own defaults are not overridden.

**Arc mark defaults:** The `"arc"` config block sets two hardcoded values not exposed as `theme()` parameters. `innerRadius = min(chartWidth, chartHeight) / 4` makes all arcs donuts by default (25px at 100×100). `padAngle = 0.03` (radians) adds a uniform gap between slices. Both are always present; override per-chart via `mark_arc(innerRadius=0, padAngle=0)` if a full pie is needed.

**Offset defaults (`axisOffset` / `legendOffset`):** both default to `None` in `_BUILTIN_DEFAULTS` and are **resolved once** in `theme()`'s computed-defaults block to `tickSize * 1.5` (`4.5` at the default `tickSize=3`) — a Prism-style detached-axis gap large enough to read as intentional rather than a rendering artifact. Resolving in one place (not inline at each use) means `config.axis.offset`, `config.legend.offset`, and `save()`'s grid-span fix all read the same resolved number from `alt.theme.options`; the axis config still zeroes it under `closed` (`0 if closed else opts["axisOffset"]`), and `legendOffset` is unaffected by `closed`. An explicit value (kwarg or TOML) skips resolution and is used as-is. After `theme()`, both are always concrete numbers in `alt.theme.options` (never `None`), so the baked theme / `read(what="theme")` records the resolved value.

**`viewPadding` (closed-plot data inset, 2026-07-11):** `viewPadding: float | bool = True` in `_BUILTIN_DEFAULTS` (the `float | bool` pattern of `cornerRadius`/`boxplotOutliers`; named as viewFill's sibling - it pads inside the view) -> `config.scale.continuousPadding`, emitted ONLY when truthy AND `closed` (an open plot's detached axes already give the marks room; `inwardTicks=True` implies closed). `True` (default) emits a 1px request = the MINIMAL effective inset; `False`/`0` -> flush (key omitted, Vega-Lite defaults). **Vega-Lite quantizes it:** `continuousPadding` extends the DOMAIN by the pixel-equivalent and then nice-rounds it outward, so ANY request is a FLOOR landing on the next nice tick boundary (verified: 2/4.5/6px requests rendered identically at ~9px on a [0,10] domain) - a 1px request buys exactly one nice step, and exact pixel gaps are impossible. Even a 1px request extends a domain bound to the NEXT nice increment (verified: interior data with extent [0.12, 9.4] went [0,10] -> [-1,10] under a 1px request; NOT only when data touches the boundary), so the True default visibly changes most closed plots by one nice step - that is the smallest inset Vega-Lite can produce, per the user's explicit choice (minimal-by-default). (A precise-gap alternative - detaching the closed frame+axes outward by axisOffset via an SVG fixer, `offsetView` - was built and REJECTED 2026-07-11; user decision, do not reintroduce.) The violin's absolute-x layer and `add_labels`' pinned x/y scales pin `padding=0` (their pixel math assumes the full [0, chartWidth/Height] range; the pin wins the shared scale, so labelled charts render unpadded by design). Tests: `TestViewPadding` in `tests/test_theme.py`.

**Gradient-legend titles (feature removed 2026-07-10, never shipped):** gradient legend titles render at Vega's default — horizontal, on top, same as symbol legends. The `legendTitleGradientOrientation` theme option (+ its save/show-time spec transform `export._orient_gradient_titles`, which injected `legend.titleOrient` into continuous color/fill/stroke encodings) merged to main post-v3.6.0 (PR #92) and was **removed before any release** per user decision — the rotated matplotlib-colorbar style was tried and rejected. No compatibility consequence: no released version ever baked the key into exports. The compact legend title/label spacing from the same PR (2px label / 4px title gaps, see the chrome-text-offsets point) was KEPT. Do NOT reintroduce the rotation via `config.legend.titleOrient` — that would rotate SYMBOL legend titles too (the reason the feature needed a spec transform in the first place). Safety net: `TestGradientLegendTitles.test_save_does_not_inject_title_orient` in `tests/test_export.py`.

**Chrome text offsets (label 2 / title 4):** the axis + legend chrome uses one consistent spacing hierarchy - a **2px label gap** and a **4px title gap** - matching Vega's axis defaults, applied to the legend too. Axes keep the Vega defaults (`config.axis.labelPadding` 2 = tick mark → tick label; `config.axis.titlePadding` 4 = tick labels → axis title), so those are NOT set in the theme. The legend IS set explicitly to mirror them (its Vega defaults differ): `config.legend.titlePadding` 4 (title → content; default 5), `config.legend.labelOffset` 2 (symbol → label; default 4), `config.legend.gradientLabelOffset` 2 (gradient bar → its numeric labels; default 2). **`labelOffset` does NOT reach gradient labels** - the gradient bar→label gap is a SEPARATE Vega property (`gradientLabelOffset`), so both must be set. Applies to every legend (symbol + gradient). NOT offsets and left alone: `tickSize` (3, the tick-mark LENGTH) and `axisOffset`/`legendOffset` (the Prism-style detached-axis gap, `tickSize*1.5`).

**Axis element toggles:** `xAxis` / `yAxis` are toggles — setting either to `False` hides the axis domain and ticks but not labels. `xDomain` and `xTicks` (and their `y` counterparts) are ANDed with `xAxis` in `_dysonsphere_theme()`; `xLabels` / `yLabels` are independent and not gated by `xAxis` / `yAxis`. For `ticks`, the global `ticks` parameter is additionally ANDed in. This lets you e.g. hide only tick marks (`xTicks=False`) or only labels (`xLabels=False`) while keeping the domain line.

**Exact tick positions (`tickRound` / `axisBand`) — the source-level alignment fix:** Vega rounds axis tick/grid SVG positions to integers for on-screen crispness while marks stay fractional, which at high DPI (1200 PPI) drifted ticks visibly off their marks. Through v2.0 this was patched *after* rendering by ~400 lines of SVG surgery (`_fix_tick_alignment` anchor-reading + band-formula fallbacks, `_fix_log_minor_ticks` interval math — both deleted). It is now fixed **at the source** by two hardcoded theme config entries (not user params, like the `"translate": 0` entries): `config.axis.tickRound: false` (disables the integer rounding) and `config.axisBand.tickOffset: 0` (kills the extra 0.5px band-scale tick offset — **must go through `axisBand`**, the scale-type-specific config; `config.axis.tickOffset` is ignored for band axes). Result, verified empirically: ticks land on the exact fractional scale position — identical to the mark position to the last float digit — for band centres (boxplot/strip/violin), linear axes, heatmap bin edges, log/power minor ticks, and every panel of hconcat/vconcat, with no post-processing and no per-mark-type heuristics to break. Because it's *config*, it travels with the spec: vl-convert SVG/PNG, `format="html"` (browser Vega), and bare-chart notebook previews all render exact ticks. Trade-off: fractional positions mean hairline ticks are slightly soft at 1:1 screen zoom — irrelevant at print DPI, and no different from the post-fixed output before. Safety net: `TestExactTickPositions` in `tests/test_export.py` asserts tick == mark equality (`abs=1e-6`) in real rendered output for the band, linear, log-minor, and mixed-hconcat cases.

**`inwardTicks` (physics/astronomy tick direction):** `inwardTicks: bool = False` in `_BUILTIN_DEFAULTS`. `True` makes axis ticks point INTO the plot and **defaults `closed=True`** (resolved in the computed-defaults block: `p["closed"] = p["inwardTicks"] or p["viewFill"] is not None` — an explicit `closed=False` still wins), which yields `axisOffset=0` (a boxed frame — the physics look). **NOT implemented as a negative config `tickSize`:** Altair's schema rejects a negative `tickSize` on encoding axes AND `configure_axis` (min 0); a negative value in the raw *theme config* survives `to_dict()`, but the config travels into `format="html"`, where the browser's Vega build mangles axis-label spacing with a negative `tickSize` (see the export.py module note) — so it stays an SVG post-process. `export._flip_ticks_inward(svg)` negates the non-zero `x2`/`y2` of every `<line>` inside a `role-axis-tick` group at `save()` time (before `_simplify_svg`) — flipping primary, secondary (right/top when explicitly enabled), major, AND log/power minor ticks uniformly, with NO mirroring on the opposite axes (x-axis ticks carry length in `y2`/`x2="0"`, y-axis in `x2`/`y2="0"`, so negating the non-zero coordinate flips direction). The fixer ALSO pulls each axis's labels + title toward the view by that axis's OWN tick vector (read pre-negation, so per-axis lengths work - log/power minor-tick axes are half-size but carry no labels): Vega placed them beyond the outward tick, so the flip would otherwise leave a dead `tickSize` gap between the domain and the labels (fixed 2026-07-11). The shift is baked into each `<text>`'s own leading `translate` (rotation tails preserved), NEVER a group transform - `_simplify_svg` drops group transforms when flattening. Axes without tick lines are untouched. Because it's a save-time SVG step, inward ticks render via `ds.save()`, not the interactive notebook preview. Gated by `if alt.theme.options.get("inwardTicks")` in the render loop. Tests: `TestInwardTicks` in `tests/test_theme.py`, `TestFlipTicksInward` in `tests/test_export.py`. Demo: `scripts/plots/inward_ticks.py`.

**Label angle parameters:** `xLabelAngle` (default `0`) and `yLabelAngle` (default `0`) replace the old `angledX`/`verticalY` booleans. Both accept any float in degrees; `labelAlign` is auto-derived from the sign: for `axisX`, negative → `"right"`, positive → `"left"`, zero → `"center"`; for `axisY`, non-zero → `"center"`, zero → `"right"`. Values are wrapped with `% 360` before being passed to Vega-Lite. Both `mark_violin` and `mark_strip` expose their own `xLabelAngle: float | None` parameter that reads from `alt.theme.options["xLabelAngle"]` when `None`.

**Axis title parameters:** Both `mark_violin` and `mark_strip` accept `yTitle: str | None = _UNSET` (defaults to `yCol`) and `xTitle: str | None = _UNSET` (defaults to `xCol`). These are forwarded to the `alt.X`/`alt.Y` `title=` argument. The `_UNSET` sentinel distinguishes "user explicitly passed `None`" from "user did not pass anything" (the latter uses the column name as the title). `mark_strip` accepts `markSize` and `markOpacity` (renamed from `pointSize`/`pointOpacity`); both fall back to `markSize` and `markFillOpacity` from the theme when `None`.

**`errorbar` config in `_dysonsphere_theme()`:** The `"errorbar"` block sets rule and tick properties independently of the global `"rule"` config. `"thickness": opts["markStrokeWidth"]` controls the stem stroke width; `"rule": {"strokeDash": [0, 0], "strokeWidth": opts["markStrokeWidth"]}` explicitly overrides the global rule config (which defaults to `axisWidth`) so stem and caps match. Tick caps use `"thickness": opts["markStrokeWidth"]` and `"cornerRadius": opts["markStrokeWidth"] / 2` (half the cap height) for rounded ends. All three (stem + rule + tick thickness) are `markStrokeWidth` (not `* 2`, as through v3.1.0) so error bars weigh the same as every other mark stroke and the axis line (`markStrokeWidth = axisWidth`). `strokeDash` is hardcoded `[0, 0]` — errorbars do not inherit `dashedRule` because dashed error bars are almost never desired.

**Boxplot proportions & default fills:** the `"boxplot"` config sizes (relative to `markSize`): box `size = markSize * 0.9`, median `size = markSize * 0.9` (spans the box), whisker end-cap ticks `size = markSize * 0.45` (**half the box width**), whisker line `rule.size = markSize` (INERT — a boxplot whisker's rendered thickness is set by `rule.strokeWidth`, not `size`; verified). All boxplot strokes (`box`/`rule` `strokeWidth`, `ticks.thickness`, median `thickness`) use `markStrokeWidth`. The **median is a single stroke**: `thickness = markStrokeWidth` with fill `markMedianFill` and NO outline `stroke`/`strokeWidth` (a competing stroke swamps the thin fill and reads as a dark sliver). **Box fill: the box has NO explicit `fill`** in `config.boxplot.box` — a boxplot box is a Vega-Lite `bar` mark, so it inherits `config.bar.fill = opts["markFill"]` (this leak is inherent to how VL lowers composite marks). Default `markFill = colors["greys"][1]` (`#DBDBDB`, a light grey that reads on both light/dark backgrounds — changed from `"black"`), and `markMedianFill = "black"` (contrasts the grey box). A `color=` encoding overrides the box fill via `config.range.*` and bypasses `markFill` entirely.

**`config.circle` defaults (raw-point overlays):** `mark_circle` is primarily used to layer raw data points over boxplots/violins/strips, so its config is tuned for small overlay dots: `size = markSize / 20` (0.5 area at the default `markSize=10` — much smaller than `config.point.size = markSize / 2` or `config.square.size = markSize`), `fill = "black" if not darkmode else "white"` (a solid dot that contrasts the background, darkmode-aware like the boxplot median), and `stroke = None` (no outline — at this dot size a stroke swamps the fill; explicit None so nothing is inherited). `strokeOpacity`/`strokeWidth` stay configured so a re-enabled stroke renders with the house style. **`mark_strip`'s points are also `mark_circle` marks but PIN their own stroke in the mark call** (`"black" if darkmode else markStroke`, house width/opacity) — the outlined-dot strip/beeswarm look must not silently follow this config (that regression shipped briefly on 2026-07-05). `mark_strip` sets its own circle size via its `markSize` param and is unaffected by this config.

**`config.tick` (crossbar defaults, 2026-07-11):** the theme styles bare `mark_tick` as a house crossbar - darkmode-flipped `color` (`white`/`black`), `cornerRadius = markStrokeWidth/2` (round caps) and `thickness = markStrokeWidth` (all three mirroring `config.errorbar.ticks`), `size = markSize*0.9` (the `config.boxplot.median` span), `opacity = markFillOpacity` - so `base.mark_tick().encode(y="mean(v):Q")` composes with `base.mark_errorbar(extent="stderr")` as one glyph with nothing pinned (the documented DIY errorbars+mean recipe; `mark_strip`'s mean tick inherits from this block too). For per-observation strip dashes, override `size` per chart. The rounded corners change the SVG tick path from `M…h…` to `M…L…C…` - the exact-tick capstone test parses both. `tick.size`/`median.size` can't reference each other in the emitted config (static JSON, no cross-references); both derive from the same `opts["markSize"] * 0.9` expression, and `TestTickConfig.test_mirrors_errorbar_caps_and_median_span` (`tests/test_theme.py`) fails CI if they drift. **Leak-through and the explicit pins:** a boxplot's median and whisker caps are tick marks internally, and the composite lowering lets `config.tick` supply any property their part configs leave unset (same mechanism as the `config.bar.fill` box leak) - so `config.boxplot.median` pins `cornerRadius: 0` (square, box-flush ends; config.tick's round caps would otherwise reach it) and both `config.boxplot.median` and `config.boxplot.ticks` pin `opacity: 1` (config.tick's `opacity = markFillOpacity` would otherwise MULTIPLY with their `fillOpacity` - double-dimming at non-default `markFillOpacity`). With the pins, boxplot/violin/errorbar renders are pixel-identical to pre-config.tick (the median gains only an inert explicit `opacity="1"` attribute, the SVG default); axis ticks are axis chrome, never mark config, and are untouched. Guard: `TestTickConfig.test_boxplot_median_pins_square_corners`.

**`config.trail` (2026-07-11):** `trail` was the last colour-bearing bare mark without a theme block (`image` has nothing themeable - it renders bitmaps), so `mark_trail` rendered raw steel blue. The block sets darkmode-flipped `color` (trail is a FILLED variable-width path - colour supplies the fill), `opacity: 1`, and `size = axisWidth * 1.5` (`config.line`'s strokeWidth, so an UNSIZED trail renders exactly like a line; `TestTrailConfig` asserts the equality). No `strokeDash` - trail's variable-width geometry can't dash. No composite lowers to trail, so there is no leak surface. **Gotcha: a continuous colour encoding on a trail draws NOTHING** - Vega-Lite GROUPS the trail by the colour field, so a Q field degenerates to single-point trails (lines get gradients; trails can't) - colour multiple trails by a nominal id with a sampled palette range (`ds.palette("australis", n)`) instead.

**`boxplotOutliers` param (`float | bool = False`):** controls `config.boxplot.outliers.size`. `False` (default) → `0` (outliers hidden — dysonsphere overlays raw data via `mark_strip`/beeswarm instead, so separate outlier glyphs would double-draw). `True` → resolved in `theme()` to `markSize / 10` (the `is True` identity check, like `cornerRadius`). An explicit number → used as the size. Wired as `"size": opts["boxplotOutliers"] or 0` (so `False`/`0` → hidden). The outlier glyph is otherwise fully styled in the config (darkmode-aware `color`/`fill`, `markStroke`); only the size gates visibility. Per-chart `mark_boxplot(outliers={"size": n})` still overrides the config.

**Boxplot tick cap rounding:** The `"boxplot"."ticks"` block also sets `"cornerRadius": opts["markStrokeWidth"]`. Unlike errorbar ticks (which now use `cornerRadius = markStrokeWidth / 2`, half the cap height), boxplot tick caps use `cornerRadius = markStrokeWidth` at `thickness = markStrokeWidth`, so the radius equals the full thickness — a more aggressively curved end. Both use rect marks; `strokeCap` has no effect on filled rects, `cornerRadius` is the correct property.

**`axisRight` / `axisTop` config:** Both are fully specified in `_dysonsphere_theme()` to mirror their counterparts (`axisY` / `axisX`) with two modifications: `labelAngle` is negated (`(-angle) % 360`) to maintain visual symmetry on the opposite edge, and `labelAlign` is flipped accordingly (`axisRight`: `"left"` at zero instead of `"right"`; `axisTop`: negative angle → `"left"` instead of `"right"`). All other properties (`domain`, `labels`, `ticks`, `translate`) are inherited directly from the same toggles as the primary axis, so secondary axes obey `xAxis`/`yAxis` visibility settings.

**`save()` format/background defaults (v1.1):** `save(format=None, background=None)` — both accept a **string or list** and default (when `None`) to the theme options `saveFormat` (`_BUILTIN_DEFAULTS` `["svg", "json"]`) / `saveBackground` (`"light"`), so the export defaults are TOML-configurable like any theme param. Resolved + validated up front by `_resolve_choice(value, default, valid, name)` (normalizes str→list, raises `ValueError` on empty or unknown values) **before** draining the report queue, so an invalid call errors cleanly and leaves the queue intact. **Filename scheme:** a `_light`/`_dark` suffix is added **only when `len(backgrounds) > 1`** (`_path(bg, ext)` helper); a single-background export writes clean names (`fig.svg`, `fig.json`, `fig.png`) — and the JSON dropped its old `_vegalite` infix. **`saveVegaSpec` was removed** (folded into `format` — `"json"`). The render loop is per-background: JSON and HTML are written at the background's `darkmode` with `transparent` at its logical (original) theme value; SVG/PNG render at the **`save(transparent=)` param** (default `True` - transparent exports always, opt out per call with `transparent=False` which fills `chartFill`; there is deliberately NO theme-level default for it, so a TOML file can never silently make saves opaque). The metadata block is **rebuilt per background** (`metadata._build_block` re-reads `alt.theme.options`), so each variant's baked `theme.darkmode` matches how it rendered (fixes an old latent bug where all variants shared the original darkmode). The SVG is a **transient PNG source** when `"png" in format` but `"svg"` isn't — it's rendered, used, then `unlink()`ed. `vl_convert` is only imported when svg/png is requested (json-only skips rendering entirely).

**`transparent` (renamed from `transparentBackground`, v3.0) + `chartFill` auto:** the theme option `transparent` is the chart's LOGICAL background switch (`config.background = None if transparent else chartFill`), affecting bare notebook display, JSON, and HTML; the `notebook` preset sets it `True`. The old name `transparentBackground` (last shipped v2.0.0) was **renamed to `transparent` as a v3.0.0 breaking change**; the dev-only alias that briefly mapped it (`_DEPRECATED_ALIASES` + `_apply_deprecated_aliases()`) was **deleted in the v3.0.0 release step-0 sweep** and never shipped in a release. Consequence: files exported by v2.x bake the old key in their theme block, so `load(applyTheme=True)` on them **raises** a clear `TypeError` naming `transparentBackground` (use `raw=True` or re-export; user accepted this). Safety-net test: `TestReadLoad.test_load_rejects_removed_theme_key` (asserts the raise + the `raw=True` workaround). **`chartFill=None` means auto** - resolved at CONFIG-BUILD time in `_dysonsphere_theme()` (`black` if darkmode else `white`), NOT in `theme()`'s computed-defaults block, because `save()` toggles `darkmode` per background variant without re-running `theme()` - the fill must follow darkmode live, like every other darkmode-aware colour (a theme()-time resolution froze the light fill into dark renders). An explicit `chartFill` is used as-is in both modes.

**`save()` `maxRows`/`overrideMaxRows` (v1.1):** `maxRows: int = 5000` (Altair's default) caps the rows inlined for the save; the render loop runs inside `alt.data_transformers.enable("default", max_rows=None if overrideMaxRows else maxRows)`, managed by an `ExitStack` (`_cap_stack`) so it restores on the way out (even on error). **All formats** enforce it — every format renders via `to_dict()`, which inlines the data (and the JSON embeds it for `read(what="data")`) — so over the cap Altair raises `alt.MaxRowsError`, caught in the loop's `except` and re-raised as a **clear `ValueError`** pointing to `maxRows=`/`overrideMaxRows=True`. `overrideMaxRows=True` → `max_rows=None` (unlimited). The `enter_context` call needs `# ty: ignore[invalid-argument-type]` (Altair's `PluginEnabler` lacks a context-manager stub). Tests in `TestSave` (test_max_rows_blocks_large_data / _raised_allows / _override / _restores_transformer).

**Dark mode:** `save()` temporarily toggles `darkmode` in `alt.theme.options` per background variant, then restores the original value in a `finally`.

**`description` (v1.1):** The `description=` param is purely the user's text — stored verbatim, nothing appended. It's set as the chart's `description` property in `_resolve()` (→ Vega-Lite JSON `"description"` key), injected as `<desc>{html.escape(description)}</desc>` into the SVG (via `re.sub` after `_simplify_svg`, since Vega-Lite's SVG renderer doesn't emit `<desc>` from the property), and written as the PNG `iTXt Description` chunk (`_inject_png_metadata()`). Independent of `saveMetadata`. When `description=None`, none of the three get a description/`<desc>`. **It ALSO rides inside the structured `dysonsphere` block (when `saveMetadata`) as `usermeta.dysonsphere.description`** — passed into `_build_block(description=)`, placed as the **last member** (after `report` in the JSON `usermeta`; last in the report-free SVG/PNG structured blob), so it's machine-readable via `read(what="metadata")` from all three formats. Deliberate duplication: the native `description`/`<desc>`/`iTXt Description` is the presentation channel (generic tools read it), the block copy makes it a first-class member of the self-contained metadata. Tests: `test_description_in_structured_block_all_formats` / `_is_last_after_report` / `test_no_description_key_when_none` in `TestSaveUsermeta`.

**Structured metadata block (v1.1):** When `saveMetadata=True`, `save()` builds `_ds_block = {"provenance": {...}, "statistics": [...]}` (statistics omitted when no records) and embeds it — **as structured JSON only, never as prose** (dropping the old "Generated by…" sentence and the report text avoids file bloat + duplication; the prose report stays available via `add_comparisons(report=True)`/`save=`). Embedded in **all three** formats so each is self-contained: (1) **Vega-Lite JSON** `usermeta.dysonsphere` — attached in `_resolve()` via `.properties(usermeta=...)` and **merged** into any user-set `usermeta` (`{**existing, **{"dysonsphere": block}}`, read via `getattr(c, "usermeta", alt.Undefined)`); (2) **SVG** `<metadata id="dysonsphere"><![CDATA[...]]></metadata>` (`re.sub` with a **lambda** replacement so JSON backslashes/braces aren't reinterpreted as regex refs); (3) **PNG** `iTXt` keyword `"dysonsphere"` (`_inject_png_metadata(..., keyword="dysonsphere")`). Serialized as `_usermeta_json = json.dumps(_ds_block, ensure_ascii=False)` (keeps `η²`/`─` literal UTF-8). `provenance` field order is **human-first**: `user → script → chart? → timestamp → environment → vegaliteChecksum → exportIdentifier → dataChecksum` — the readable who/when/how leads (`chart` is the CONDITIONAL best-effort call-site capture, directly after `script`, its sibling source-code fact - see the provenance `chart` design point), then the machine-identity hashes trail, with the **unbounded `dataChecksum` list LAST** (it grows with multi-frame charts, so trailing it keeps the readable top of the block stable). This mirrors the `_render_provenance` prose, which also puts the checksums after the "Generated by …" sentence. `environment` is a platform + toolchain dict **`{os, python, altair, vl_convert, dysonsphere, dysonsphere-extensions?, numpy, scipy, polars}`** — `os` (from `platform.platform()`) first as the top-level spec, then the interpreter, the spec generator (`altair`), the **renderer** (`vl_convert`), the tool, and its declared runtime dependencies. Grouped so the platform + toolchain sit in one place; the top level stays identity + who/when. **`os` + `vl_convert` together pin *rendering*:** vl-convert resolves fonts partly from the system AND bundles its own Vega/Vega-Lite build, so text metrics (superscript alignment, `add_labels` box sizing) can differ across platforms and renderer versions — they're the reproducibility axes beyond the pinned lib versions. **Provenance `chart` (call-site capture, 2026-07-08):** `provenance.chart` is the **best-effort verbatim source text of the `chart` argument at the `save(...)` call site** - the variable name (`"fig"`), the inline composition (`"boxplot + points + line"`), or a whole multi-line lambda (newlines kept) - answering "what expression produced this file" next to `script` ("which file"). **`save()` provenance identity (v1.1; reordered to trail 2026-07):** three machine-identity fields sit at the **end** of `provenance` (after `environment`; the human-readable context leads instead), ordered **`vegaliteChecksum` → `exportIdentifier` → `dataChecksum`** so the one unbounded field (`dataChecksum`) is LAST: **`vegaliteChecksum`** = `"sha256:<64hex>"` of the canonical spec with `usermeta` removed and markers stripped (`metadata._spec_checksum`; sort_keys + compact separators → reproducible; re-validate by stripping `usermeta` from the file and re-hashing) — the true content identity, **per variant** (light/dark differ); **`exportIdentifier`** = a `uuid4`, **generated once per `save()` call** (with `timestamp`) so every variant/format of one export shares it; and **`dataChecksum`** = a **sorted list** of `"sha256:<64hex>"`, **one per user dataframe** (`metadata._data_checksum`) — the data identity, **order-independent** (see below). So two files with the same `exportIdentifier` came from one `save()`; the same `vegaliteChecksum` means identical spec content; the same `dataChecksum` means identical underlying data. So two files with the same `exportIdentifier` came from one `save()`; the same `vegaliteChecksum` means identical spec content; the same `dataChecksum` means identical underlying data. `_build_provenance(export_id, timestamp, checksum, data_checksum)` takes them as args (export_id/timestamp shared, checksum per-variant; data_checksum per-variant too but identical across light/dark since darkmode changes theme/config, not data); `_render_provenance` prepends the `Data checksum(s):` line(s) (one line for a single frame, an indented `- ` list for multiple), then the `Vega-Lite spec checksum:` and `Export identifier:` lines, to the prose. The JSON is written by hand (`json.dumps(spec-with-usermeta, ensure_ascii=False, indent=2)`) rather than via Altair's `.save()` so markers can be stripped from the dict first; SVG/PNG render from `base_obj` (markers present in the object but never rendered).

**`dataChecksum` design (v1.1):** the point of a *separate* data checksum is to be **more invariant** than `vegaliteChecksum` — it identifies the data a chart was rendered from independent of *how* it was drawn, so two charts with **different specs** (marks/encodings/themes) but the **same data** match. Two invariances distinguish it: (1) **order-independent** — `vegaliteChecksum` already changes with row order (the inlined data is part of the spec it hashes), so `dataChecksum` deliberately ignores it, hashing the **multiset of per-row canonical-JSON digests** (`sorted(sha256(json.dumps(row, sort_keys=True, separators=(",",":"), ensure_ascii=False)) …)` → hash the sorted digest list). Robust (sorts hex strings, never heterogeneous row values); duplicate rows preserved (multiset, not set). (2) **spec-independent** — only the **user's** frame(s) are hashed; internal sidecar datasets (brackets/KDE/error-bars/annotations, tagged with `utils._INTERNAL_COL`) are excluded, so adding an annotation layer does not change it. Both `_data_checksum` and `read(what="data")` collect user frames via the shared **`metadata._user_datasets(spec)`** helper (named datasets + inline `data.values`, sidecars filtered) — one source of truth. Multi-frame charts (`hconcat`, …) yield a **sorted list** (one entry per frame; the list itself sorted so frame order doesn't matter). Tests: `test_data_checksum_*` in `TestStatsQueueRobustness` (matches-across-specs, differs-for-different-data, order-independent, shared-across-variants, excludes-internal-sidecars, multiframe, revalidates).

**`add_rule()` design:** Lives in `annotations.py` (`_rule_mark_kwargs` private helper). Unified horizontal/vertical reference line function; `axis="y"` (default) draws horizontal lines at fixed y value(s), `axis="x"` draws vertical lines at fixed x value(s). **Both** the default and the `data=` (facet-safe) modes build their layers via `_datum_ref_layers()` — one `mark_rule` layer per value (+ one `mark_text` label layer per value when `label` is set), each positioned by a constant **`alt.datum(v)`**, NEVER a data field. **Returns a bare `alt.Chart` for a single unlabeled value; an `alt.LayerChart` for multiple values or any labels** (one datum layer per value can't collapse into one). **Why datum, not a `__v` field (the title-clobber fix, 2026-07-05):** a field on the shared position channel participates in Vega-Lite's layer axis-title merge — an explicit `title=None` NULLS the base chart's axis title, and a derived field title (`"__v"`) concatenates INTO it (`"weight, __v"`); both are wrong. A constant `alt.datum` has no field, so it contributes no title and the base axis title survives. (This replaced the old field-based default path, whose `title=None` silently stripped the base title — verified against the reference-line/text gallery examples.) The two modes differ only in the per-layer base (`_datum_ref_layers(base_factory, …)`): `data=` mode uses `_datum_base(src)` (shares the frame → faceteable); the default uses a fresh `alt.Chart(_internal_data([{}]))` sidecar (read-filtered, deliberately NOT faceteable). The `_rule_mark_kwargs()` private helper builds the `mark_rule` property dict (color, strokeWidth, strokeDash, opacity). `strokeDash=None` (default) does not pass `strokeDash` to the mark so the theme's `dashedRule` config applies automatically; `strokeDash=False` passes `[0, 0]` to force solid; `strokeDash=True` reads `dashedWidth` from `alt.theme.options`. For labeled lines: `axis="y"` places text at `x=alt.value(0)` (left edge) or `alt.value(chartWidth)` (right), using `chartWidth` from `alt.theme.options`; `axis="x"` places text at `y=alt.value(0)` (top) or `alt.value(chartHeight)` (bottom). Scale sharing is implicit — Altair layers share quantitative scales by default, placing reference lines at the correct data coordinates on the main chart's scale. Label geometry (perpendicular anchor + `mark_text` kwargs) is resolved once by `_rule_label_geometry()` and shared by all layers so their placement can't drift. **Closed-plot edge inset:** an edge-anchored label (left/right for `axis="y"`, top/bottom for `axis="x"`) is inset from the content edge by `axisOffset` **when the plot is closed** (`edge_inset = _opt("axisOffset") if _opt("closed") else 0`, added to the left/top anchor and subtracted from the right/bottom). An open plot's spine is detached by `axisOffset`, so a label at the content edge (anchor `0`) already clears it; a closed plot's spine is flush at the content edge, so the same label would hug the border - the inset gives it the identical gap, so opened and closed look the same. Center anchors are far from either edge and untouched. Tests: `TestRuleLabelInset` in `tests/test_annotations.py`. Title-preservation regression tests: `TestAddRule.test_preserves_explicit_base_axis_titles` / `test_preserves_derived_base_axis_title`. **The same title-clobber was fixed in `add_text` and `add_shade` too** (all three `add_*` sidecar annotations now position by datum/value, never a `title=None` field).

**Facet-safe datum mode — DISCIPLINE (v1.1):** every composable `add_*` annotation builds its own `_internal_data` sidecar layer(s), which makes the composition **un-faceteable**: Altair requires all layers of a faceted chart to share ONE data variable, so a layer with its own dataset makes `.facet()` raise ("Facet charts require data to be specified at the top level") — and the `.facet(data=df)` workaround silently FAILS to partition (every panel shows the full data). The remedy is an opt-in **`data=` parameter** (named to match `alt.Chart(data=)`; the user passes the **same frame as the base chart**) that builds the annotation on **`_datum_base(src)`** (in `annotations.py`) instead of `alt.Chart(_internal_data(...))`. `_datum_base(src)` = `alt.Chart(src).transform_aggregate(__dsagg="count()")` — it (a) **shares `src`** so faceting partitions correctly, and (b) the dummy aggregate **collapses N rows → 1** so constant marks don't overplot N times. Positions must come from **`alt.datum(v)`** (data coords — works for both Q and N/band) or **`alt.value(px)`** (pixels), **NEVER a data field** (a field reintroduces a sidecar and breaks faceting); text via `text=alt.value(...)`. Multi-value → one datum layer per value. Altair dedupes the shared frame to one dataset, so `read(what="data")` is unaffected (no phantom user frame). `data` accepts polars or pandas (`ensure_polars`). **APPLY to** annotations whose content is a FIXED, data-independent reference meant to repeat identically in every panel: **`add_rule`** (`_datum_ref_layers`), **`add_text`** (per-annotation datum/value layer), **`add_shade` positions mode** (`_shade_rect` switches field↔datum on `datum_mode`). **DO NOT apply to** `add_comparisons`/`add_correlation` (their content is COMPUTED per-data → a single datum repeated across panels would be statistically WRONG; correct faceting there = per-panel recomputation, a separate feature) or **`add_shade` BAND mode** (`data=` raises there — many band rects keyed to per-panel band math; deferred). **RECIPE for a NEW facet-safe annotation:** add a `data: pl.DataFrame | Any | None = None` param; when given, `ensure_polars` it and build every mark on `_datum_base(src)` with datum/value-only encodings; leave the default (sidecar) path untouched; add a `test_default_not_facetable` guard (asserts the default raises the Altair facet error) + a `test_datum_faceting_succeeds` test. Tests: `TestAddRuleDatum` / `TestAddTextDatum` / `TestAddShadeDatum` in `tests/test_annotations.py`.

**Per-annotation datum notes:** `add_rule` — `_datum_ref_layers()` builds one datum rule (+ optional datum label) per value; label geometry via the shared `_rule_label_geometry()`. `add_text` — `_text_datum_layers()` builds one datum/value layer per (text, x, y); each x/y is `alt.datum` (Q float / N str) or `alt.value` (pixel passthrough or a `position=` preset), and the string rides as `text=alt.value(t)`. **Both `add_rule` and `add_text` now position by datum in BOTH the default and `data=` modes** (they differ only in the base factory: internal sidecar vs shared `_datum_base(src)`) — this is the title-clobber fix; a field on the shared channel would null/concatenate the base axis title. `add_shade` — the unified **`_shade_rect(color, x=…, y=…)`** helper takes each axis as `None` / `("px", lo, hi)` / `("q", start, end)` and emits `alt.value` (pixel range) or **`alt.datum` (data range, BOTH default and datum modes)** — the `__<ch>s:Q`/`title=None` field path was removed (it clobbered the base axis title, same bug as `add_rule`/`add_text`; fixed 2026-07-05). **`add_rule()` label parameters:** `labelAlign` controls where *along* the line the label is anchored; `labelPosition` controls which *side* of the line it sits on. These drive different Vega-Lite properties depending on axis:

- `axis="y"` (horizontal rule): `labelAlign` is `"left"/"center"/"right"` (default `"left"`) — maps directly to Vega-Lite `align` and the x anchor (`alt.value(0)`, `alt.value(chartWidth/2)`, `alt.value(chartWidth)`). `labelPosition` is `"top"/"bottom"` (default `"top"`) — drives `dy` direction and Vega-Lite `baseline`.
- `axis="x"` (vertical rule): `labelAlign` is `"top"/"center"/"bottom"` (default `"top"`) — maps to the y anchor and Vega-Lite `baseline`. `labelPosition` is `"right"/"left"` (default `"right"`) — drives `dx` direction and Vega-Lite `align`.

`labelOffsetX` and `labelOffsetY` are additive pixel offsets applied directly to the text mark (`dx` and `dy`). Both default to `0` (flush). The perpendicular gap between label and line is a hardcoded `±3` (`base_dy` for horizontal, `base_dx` for vertical) derived from `labelPosition`; `labelOffsetX`/`labelOffsetY` add on top of that.

**`add_text()` design:** Lives in `annotations.py` (`_is_alt_value` private helper, `_TEXT_PRESETS` dict, `_text_datum_layers` builder). Places one or more text annotations at arbitrary coordinates. **Returns a bare `alt.Chart` for a single annotation; an `alt.LayerChart` for multiple** (one layer per annotation). Three coordinate modes auto-detected from Python type per value: `float`/`int` → data coordinate on the shared quantitative scale; `str` → category name on the shared band scale (band centre); `dict{"value": n}` (i.e. `alt.value(n)`) → pixel passthrough, independent of data scales. `_is_alt_value(v)` helper detects the third form as `isinstance(v, dict) and "value" in v`. **All three are positioned by `alt.datum` (Q/N data coords) or `alt.value` (pixels), NEVER a data field — the title-clobber fix (2026-07-05):** the old default path used `__x`/`__y` fields with `title=None`, which NULLED the base chart's axis title(s) on whatever axes the annotation used (a data-coord `add_text` silently stripped BOTH axis titles); `alt.datum`/`alt.value` contribute no title, so they survive. The text string itself rides as `text=alt.value(t)` (not a `__text` field). Both the default and `data=` (facet-safe) modes share `_text_datum_layers(base_factory, …)`; they differ only in the base — `_datum_base(src)` (shared frame, faceteable) vs a fresh `alt.Chart(_internal_data([{}]))` sidecar (read-filtered, non-faceteable). Regression test: `TestAddText.test_preserves_base_axis_titles`.

**`add_text()` position presets:** `_TEXT_PRESETS` dict maps 9 camelCase names (`"topLeft"`, `"topCenter"`, `"topRight"`, `"middleLeft"`, `"middleCenter"`, `"middleRight"`, `"bottomLeft"`, `"bottomCenter"`, `"bottomRight"`) to `{x_frac, y_frac, align, baseline}`. Resolved at call time from `chartWidth`/`chartHeight` in `alt.theme.options`. When `closed=True` or `axisOffset=0` in the active theme, a fixed 1 px inset (`_pad`) is applied to edge positions (x_frac=0 or 1, y_frac=0 or 1) so text clears the border. Center positions (fractions=0.5) are unaffected. Explicit `x`/`y`/`align`/`baseline` arguments override the preset for that parameter. `offsetX`/`offsetY` are additive pixel nudges passed as `dx`/`dy` to the Vega-Lite mark (positive = right/down); applied on top of any inset.

**`add_text()` baseline defaults:** Default `baseline="middle"` when no preset is used (best near symbols/rules). Presets use type-appropriate baselines: `"top"` for top-row positions (text hangs down from anchor), `"middle"` for middle row, `"alphabetic"` for bottom-row positions (reading baseline sits on anchor).

**Text background fill (`add_text` + `add_labels`, 2026-07-12):** both gain `fill: str | bool = False` / `fillOpacity: float = 1` / `stroke: str | bool = True` / `cornerRadius: float | bool = True` — a `mark_rect` chip drawn BEHIND the text (Vega-Lite text marks have NO native background, so it's a separate rect layer). Two shared helpers in `annotations.py`: **`_resolve_text_bg(fill, stroke)` → `(fill_c, stroke_c)`** — the `bool|str` pattern: `False`→`None` (off), `True`→a darkmode-aware default (fill `greys[0]` light / `greys[11]` dark; stroke `black` light / `white` dark), a str→that color; darkmode read via `_opt("darkmode")` at BUILD time like `add_shade`, so a `save()` across backgrounds needs a **callable** to re-resolve it. **`_text_bg_props(text, fs, align, baseline, dx, dy, fill_c, stroke_c, fillOpacity, cornerRadius)` → `(rect_kwargs, xOffset, yOffset)`** — the rect is PIXEL-sized (`width ≈ len·fs·0.6 + pad`, `height ≈ fs·1.4`, `cornerRadius` from the `float|bool` param: `True`→`fs·0.25`, `False`→`0`, float→px, `strokeWidth = markStrokeWidth`) positioned at the text's SAME datum/value, then recentred onto the glyphs by pixel `xOffset` (`±w/2` for left/right align) + `yOffset` (`±h/2` for top/bottom baseline) + `dx`/`dy` — **SCALE-FREE**: works for datum (data-coord) AND value (pixel) positions with no scale needed at build (the offsets are the whole trick; verified `mark_rect` honors `width`/`height` px + `xOffset`/`yOffset` `alt.value`). **The chip is GATED ON `fill`** — `bg` is built only when `fill_c is not None`, so `stroke`/`cornerRadius` (and their `True` defaults) do NOTHING until a fill is set: a bare `add_text`/`add_labels` draws no box (preserving pre-feature behavior on every existing example; a `stroke=True`-draws-a-box-everywhere default was explicitly rejected). `stroke` therefore only BORDERS the fill chip (no outlined-transparent-chip mode). **`stroke=False` must PIN `stroke: None, strokeWidth: 0`** on the rect — the theme styles `config.rect` with a black `strokeWidth 0.25`, which a `mark_rect` inherits (same config-leak as `config.bar.fill` on boxplots), so without the pin `stroke=False` would still show the leaked black border. `_text_datum_layers` (add_text) and add_labels' label loop each prepend the rect when `bg` is set (rect BEFORE text so it's behind; in add_labels it's appended AFTER the connector, so the chip visually clips the connector at its edge — the desired "connector meets the box" look). The width is a ROUGH proportional-font estimate (the same one add_labels' `sizes` uses for placement) + padding, so the chip APPROXIMATES the text extent, not exact (Vega-Lite can't measure text at build). **`add_labels()` design (auto-placed point labels):** Lives in `annotations.py`; auto-placed non-overlapping text labels with connector lines for a set of points (volcano/MA gene labels, rank-order labels, UMAP cluster labels, …). `add_labels(df, xCol, yCol, labelCol, *, labels, xDomain, yDomain, fontSize, fontStyle, color, connector, connectorColor, connectorStrokeDash, connectorGap, alwaysShowConnectors)` returns an `alt.LayerChart` (one connector `mark_rule` + one `mark_text` layer **per label**, nothing else - the scale pin rides on the first label layer's datum encodings, see self-pinning below) to compose with `+`. **`df` is the full plotted data** (same frame as the base chart); **`labels: int | list | Any | None`** selects which rows to annotate: `None` = all; an **int `n`** auto-selects `n` rows spread evenly across the plot via the internal **`_placement._sample_spread`** (deterministic farthest-point sampling - unbiased/no cherry-picking, preferred over a density-weighted random sample that would clump); a **boolean mask** (pandas/polars `Series`, NumPy array, or list of bools, one per row of `df`) selects rows **positionally** via the private **`_bool_mask(labels, n_rows)`** helper (returns a `list[bool]` when it is an all-bool array-like of the right length, else `None` → fall through), so a **non-unique `labelCol`** still picks the intended rows (`labels=df["is_hit"]`); any other **list** matches `labelCol` values (`labels=["TP53","EGFR"]`, needs a unique `labelCol`). This decouples selection from the display column, so the intended "pass the full `df`, dodge every point, label a subset" usage works even when the label column repeats — the alternative (pre-filtering `df` to the labelled rows) leaves the labels blind to the rest of the scatter and they superimpose on the unlabelled points. `_bool_mask` disambiguates a same-length list of label VALUES from a mask by requiring every element to be a `bool`. Positions come from the selected rows but the DOMAIN + obstacles are both from the FULL `df`, so labeling a subset never clips the axes AND the labels avoid every plotted point. **Intended usage: always pass the full plotted `df` and let `labels` select** - then `xDomain`/`yDomain` are just the `xCol`/`yCol` extent (inferred, never passed). They're a manual override ONLY for derived positions where the passed df isn't the plotted data (e.g. cluster centroids, whose extent isn't the scatter's). Placement is computed in PIXEL space by **`_placement._repel_labels`** (deterministic, **no RNG** so figures are reproducible - the original gripe with Altair's placement). **Algorithm = nearest-clear-spot + 2-opt** (this replaced an earlier force-directed sim, which whack-a-mole'd between long connectors and labels-on-markers): each label (placed sparsest-anchor-first) gets a **ring search** outward from its point, taking the NEAREST position whose box clears the markers (ALL plotted points, passed as `obstacles=`) and the already-placed labels - so connector length is **dynamic**: tiny where there's open space beside the point, longer only for a genuinely buried point. At each radius the angles are tried in OUTWARD order (mark − data centroid) - a **soft directionality tiebreak** leaning labels toward the margins (left mark → left label) WITHOUT forcing a longer connector. Then a **2-opt pass** swaps which label owns which slot whenever that lowers the total cost (`Σ connector length + 50·marker-overlaps + 5000·label-overlaps + w_dir·inward-distance`): by the uncrossing lemma, minimizing length removes crossing leaders for free, and the small inward penalty keeps the soft directionality after swaps. **Soft, not hard:** an earlier "outward-first" greedy (search the whole outward hemisphere before going inward) was rejected - it hard-imposed the volcano's radial shape on ALL data, lengthening leaders on general scatters (uniform/correlated/UMAP) for no benefit; directionality must stay a tiebreak/bias over nearest-clear, never a placement rule. **Force-show: never drops a label** — in an impossibly dense region labels settle at their least-overlapping positions (rationale: dysonsphere labeling is deliberate/opt-in top-N, so dropping would violate least-surprise; connectors keep even crowded labels attributable). Label/connector positions are emitted as **`alt.datum`/`alt.X2Datum` data coordinates, never fields and never `alt.value` pixels** (the exact linear inverse of the pinned pixel map, so the round-trip is lossless) — a datum contributes no axis title (no title clobber) and does not extend the shared scale domain, so the explicit pin is the only domain influence. **Self-pinning (no invisible mark, v2 of the fix):** placement runs before Vega resolves scales, so the connectors align only if the shared scale matches — `add_labels` forces that itself by attaching `alt.Scale(domain=…, nice=False, zero=False)` to the **first label layer's `XDatum`/`YDatum` encodings** (an explicit domain on one layer wins Vega-Lite's shared-scale union; verified). This replaced the earlier invisible scale-pin `mark_point`, which leaked a phantom zero-opacity element into every export — including plain-Altair saves, since it lived in the spec. Nothing invisible remains in the spec (the sole exception: the n==0 empty-selection degenerate case returns one invisible placeholder layer so `+` still composes). So the caller does NOT pin the base scale — `base + ds.add_labels(df, …)` just works. **The default domain is the full-`df` extent rounded OUTWARD to nice tick multiples via `utils._nice_domain`** (d3's `nice()` 1/2/5/10 algorithm; exactness vs Vega irrelevant — the pin makes our rounding self-fulfilling), so the axes end on round numbers and edge markers clear the border; an explicit `xDomain`/`yDomain` is used exactly as given (no nicing). Side effect: the axes retighten around the data (round bounds, but no Vega default `zero`), which is required for alignment. `xDomain`/`yDomain` are only passed explicitly for **derived positions** (label coords not in the plotted df, e.g. cluster centroids — give the plotted extent so the axes span it). **Text justification follows the connector side (dominant offset axis, aspect-aware):** when a label sits mostly beside its point (connector meets a left/right box edge) the text is **left/right-justified**, anchored at that inner edge so it flows away from the point and stays attached when edited in Illustrator (`text-anchor="start"/"end"`); when it sits mostly above/below (near-vertical connector, e.g. a label directly over its point) the text is **center-justified** (`text-anchor="middle"`) with the connector meeting the middle of the top/bottom edge, so the connector stays vertical and a symmetric edit keeps it aligned. The connector leaves an end gap at BOTH the marker and the text anchor (so it points at the dot/label rather than piercing the marker or touching the glyphs). **Gaps are UNIFORM and ASYMMETRIC (2026-07-05):** they never shrink (the old `min(connectorGap, 0.25*segment)` shrink let short connectors - the nearest-clear-spot norm - start INSIDE the marker, giving nonuniform touching-vs-gapped dots across one chart), and the two ends differ: the MARKER end is **`connectorGap`**, defaulting to the theme's `mark_point` EDGE radius + two connector stroke widths of whitespace (`sqrt(markSize/2/pi) + markStrokeWidth + 2*axisWidth`) so it clears the default point mark (and the smaller `mark_circle`) with a visible sliver of daylight, without the caller sizing it; the TEXT end keeps only the whitespace term (`2*axisWidth` - no marker to clear there; a symmetric gap opened a hole between line and label). Every term scales with its visual referent (marker radius / marker stroke / connector stroke - the connector is drawn at `axisWidth`), NO fixed px constants. Why TWO axisWidths of daylight, not one: the rule's ROUND CAP paints `axisWidth/2` beyond each endpoint and the marker stroke paints `markStrokeWidth/2` beyond its radius, so one axisWidth left only ~0.25px of TRUE painted daylight at the default theme - sub-device-pixel in PNG exports, visible or not depending on connector angle (nonuniform-LOOKING gaps from uniform geometry; measured). Two leaves ~0.5px painted daylight. `connectorGap=0` → no marker gap, or pass a float for unusually large/thickly-stroked markers (the gap can't measure the marker - the base chart isn't visible here). **Short connectors are dropped:** when the full gaps would leave less than 4 connector-stroke-widths of visible line (**`seg < connectorGap + 6*axisWidth`**, i.e. <1px of line at the default theme), it's omitted - the stub is just noise and the adjacent label is unambiguous (like ggrepel's `min.segment.length`); **`alwaysShowConnectors=True`** forces every connector drawn (sub-threshold stubs fall back to proportionally shrunken gaps). The threshold is **font-INDEPENDENT** (tied to marker/stroke geometry): an earlier `seg < fontSize` version silently dropped real 4-6px leaders when the label font was raised. Connectors **inherit the theme's `mark_rule` config** (rounded caps, `axisWidth` stroke, opaque, darkmode-aware color); label text inherits `config.text` (darkmode-aware color) at the theme's **`fontSize`** (the primary chart font; `fontSize=` overrides). Both colors resolve per render, so they track darkmode without a callable; `color` / `connectorColor` override, and `connector=False` drops the lines. **`connectorOpacity`** (`None` → inherit the theme's opaque `mark_rule`; a float 0-1 → that opacity) sets ONLY the connector mark's `opacity`, leaving the darkmode-aware color intact — so a faded leader (e.g. `connectorOpacity=0.5` to quiet the leaders under the labels) reads in both modes, unlike baking alpha into an rgba `connectorColor`. **`connectorStrokeDash`** sets the dash pattern: `False` (default) → solid `[0, 0]` (does NOT inherit the theme's `dashedRule` — leaders default solid), `True` → the theme's `dashedWidth`, a list → that pattern. Chosen over ggrepel/adjustText-style deps because Vega-Lite has no label-repel primitive — every good volcano labeler (ggrepel, EnhancedVolcano, adjustText, d3-labeler) computes placement outside the renderer, which fits dysonsphere's precompute-then-feed-Altair pattern. Tests: `TestAddLabels` (`tests/test_annotations.py`), `TestRepelLabels` (`tests/test_placement.py`).

**`mark_violin()` x-positioning:** The violin shape is built from precomputed absolute `x:Q` coordinates rather than `xOffset`. This avoids Vega-Lite merging the violin's xOffset domain with any other chart's xOffset domain in hconcat (which would squish the violin). The band centres come from `utils.band_geometry(n, scale="band")` (matching `mark_boxplot`'s actual band scale, `paddingInner = paddingOuter = bandPadding`), which differs from the xOffset/`mark_strip` variant (`scale="offset"`, paddingInner=0) — see the utils.py `band_geometry` entry for the formulas. The violin's `x:Q` encoding uses `axis=None` (not `orient="top"`) so Vega allocates no layout space for it; without this, the hidden axis would reserve ~chart_height of space above the violin panel, pushing the chart title down and misaligning hconcat titles. The layer uses `resolve_axis(x="independent")` so the boxplot's nominal x:N axis remains at the bottom.

**`_MarkScaffold` (marks.py) — composition, NOT a base class:** the shared chrome of the custom mark constructors lives in one dataclass — `ensure_polars`, the `_UNSET` title resolution, `x_axis()` (labelAngle + `labelMap`→`label_expr`), `x()`/`y()`/`color()` encodings (category sort, palette str-or-list resolution, legend flag/symbol). Each `mark_*` function constructs a scaffold at the top and keeps only its geometry/layers; **a new shared parameter is added to the scaffold once** (+ one explicit line per public signature — signatures stay explicit for typing/docs, never `**kwargs`). Marks remain plain functions returning Altair objects: the composition algebra (`+`, concat, facet) belongs to Altair, so there is deliberately NO user-facing mark class and NO inheritance hierarchy (two marks with divergent internals — KDE paths vs jitter/error bars — share chrome, not structure). The refactor was verified spec-identical against pre-refactor `to_dict()` output.

**Layers vs. marks:** `mark_violin()` and `mark_strip()` call `alt.layer()` internally to combine multiple Altair chart objects. `add_comparisons()` returns a bare layer (or stacked layers for multiple pairs) that the caller adds to their chart with `+`. Its bracket shape is controlled by `bracketStyle` (default `"bracket"` = bar + end ticks, or `"line"` = bar only) — **accepts `str | dict` (v1.1):** a string styles all brackets uniformly; a `dict` maps a pair → style for per-pair control (keys matched by `frozenset` so pair order doesn't matter; pairs absent from the dict fall back to `"bracket"`; dict values validated to `line`/`bracket`). Resolved once into a `pair_styles` list at the top of the `if pairs:` block and passed per-pair to `_pvalue_layer(bracket_style=...)`; the auto-`yPad` target uses the larger `10 px` gap if *any* pair is a `bracket`. This is the project's second dict-or-scalar per-item style param (cf. `add_multilabel` `rowStyles`). Its label format is controlled by `labelStyle` (`"p"` for `P = 0.012` / `P < 0.001`, or `"asterisks"` for `*` / `**` / `***` / `ns`). **`labelStyle` affects only the pairwise brackets** — the omnibus/test label (`_omnibus_label`) always renders the p-value (`_format_pvalue`), never asterisks (`Kruskal-Wallis ***` reads oddly). `notation=` controls the number format when `labelStyle="p"`: `None` (default, preserves current behavior) / `"scientific"` (`P = 1.23×10⁻⁵`) / `"e"` (`P = 1.23e-05`) / `"power"` (`P ≈ 10⁻⁵`, rounds to nearest power — values within the same decade get the same label). `"si"` raises `ValueError`. **`notation` also accepts `str | dict` (v1.1)** — same per-pair schema as `bracketStyle` (tuple keys matched by `frozenset`, unlisted → plain `None`, values validated) PLUS a special **`"test"` string key** for the omnibus/test label's notation (named `"test"` to match the `testLabel*` param family, NOT `"omnibus"` — user decision; unknown string keys raise `ValueError`). Resolved early (before the test-label block, so `test_notation` is available for `_omnibus_label`) into `pair_notations` (per bracket) + `test_notation` (omnibus label); a dict with no `"test"` key → omnibus label plain. This is why the resolution is at the method-resolution point, not in the `if pairs:` block (also lets `ty` narrow the scalar branch). **`sigFigs` replaced `decimals` (v1.1)** — precision is now significant figures, not decimal places, so it's consistent across magnitudes (`sigFigs=2` → both `4.3×10⁻¹⁴` and `0.68`). It's a `theme(sigFigs=3)` global + per-call `sigFigs=None` (None reads the theme) on `add_comparisons`/`add_correlation`, resolved to `effective_sigfigs` and passed to `_format_pvalue`/`_omnibus_label`/`_correlation_label`. Implemented via `%.{n}g` (strips trailing zeros — `P = 1`, not `1.000`). The `P < 0.001` floor is now a **fixed convention** (was `10^(-decimals)`), only for plain `notation=None`; scientific/e/power never floor. Also governs the correlation readout numbers (`r`/`r²`/slope/intercept — were hardcoded `.2f`).

**p-value label vertical offset:** `_pvalue_layer` (private) uses `dy = ±4` for alphanumeric labels (`"P = ..."`, `"ns"`) and `dy = ±2` for pure asterisk glyphs (`*`, `**`, `***`). `reverse=False` (text above the bar) keeps the **inherited default baseline** (its spacing already looks right — don't set one, or it shifts). `reverse=True` (text below the bar) sets **`baseline="top"`** so the text hangs below the bar instead of overlapping it (the inherited baseline left the below-bar text cramped). Only set the explicit baseline for the `reverse=True` case. The `tickHeight` parameter defaults to the theme's `tickSize` (px → data units via `tickSize * y_range / chartHeight`) so bracket end ticks match the axis ticks; `y_range`/`chartHeight` are computed unconditionally (not just when `yPad is None`).

**p-value `yPad` auto-scaling (v3.3):** `yPad` defaults to `None` and is computed as `target_px * y_range / chartHeight`, targeting a ~8 px visual gap for `bracketStyle="line"` and ~10 px for `bracketStyle="bracket"`. **`y_range` is the FULL data extent (all groups), not just the compared groups.** This matters because bracket positions are emitted in *data units* on the shared y-scale, but the visual gap is `yStep * chartHeight / rendered_domain` — and Vega fits `rendered_domain` to every group. Sizing `yPad` off only the annotated groups collapses the brackets to a sub-pixel sliver when an un-annotated group (e.g. a saturating positive control) inflates the domain; using the full extent makes `y_range ≈ rendered_domain`, so the gap stays stable across charts and never collapses. `yStep = yPad * 1.75` cascades from this (the stacking gap between levels; `1.75` chosen so a bracket's label clears the bracket above it without the airy look `2.0` gave in 3.2.0 on charts whose rendered domain hugs the data extent). **`yStart` still anchors to the *annotated* groups' max** (`max(annotated) + yPad`), so brackets sit above the compared data, not up at the dominating group's level. This is a domain-*dependent* heuristic by necessity: the gap can't be made pixel-exact without pinning the y-domain (rejected - it compresses the data), predicting Vega's domain (fragile), or a two-pass render (doesn't fit the `+`-composition model); Vega-Lite also can't express "data position + pixel offset" on a rule's `y`. Peer tools (statannotations/ggsignif) use the same data-unit-factor approach. Residual limitation: an explicit wide base `domain=` (data doesn't reach the top) still under-scales, since `df` can't see the base chart's forced domain - covered by explicit `yStart`/`yStep`. `tickHeight` is independent (defaults to the theme's `tickSize`, converted to data units via the same full `y_range`). Explicit `yPad`/`yStep`/`yPositions` override the auto values.

**Font-size params `secondaryFontSize` / `smallestFontSize` (v1.1):** `secondaryFontSize` is a derived param (`_BUILTIN_DEFAULTS` value `None`, computed in `theme()` as `max(1, fontSize - 1)` — `6` at the default `fontSize=7`, scales with `fontSize`, clamps at `1`), then **floored at `smallestFontSize` when `fontSize >= smallestFontSize`** (so it never dips below the floor in normal use; an explicit `fontSize` below the floor is the escape hatch that disables the floor and lets the tier follow the small base). `smallestFontSize` is a fixed floor (`_BUILTIN_DEFAULTS` value `5`, NOT derived) that doubles as a minimize switch, typed `int | bool`: `True` → `fontSize` is set to `5` (and `smallestFontSize` resolves to `5`) so the whole plot shrinks to the floor — and because of the floor rule `secondaryFontSize` becomes `5` too (not `4`); `False` → resolves to the int `5`; an `int` → used as-is (retrievable, floors `secondaryFontSize`, no minimize). The `is True`/`is False` checks (identity, not `==`) keep `smallestFontSize=1` an int, not a switch. After `theme()`, `smallestFontSize` is always an int in `alt.theme.options`. **All in-plot annotation labels (`add_comparisons` bracket p-values + omnibus/test label, `add_correlation` readout) default their `fontSize` to the primary `alt.theme.options.get("fontSize", 7)`** (reverted from `secondaryFontSize` per user decision — stat labels should match the axis font). So `secondaryFontSize` is currently NOT consumed by any rendering code (it's an exposed auxiliary param for users/future); `smallestFontSize` is consumed only as the floor on `secondaryFontSize` and as the minimize switch. Per-call `fontSize=` still overrides annotation sizes.

**p-value `reverse` + negative `yStep`:** To place brackets below the data, set `yStart = data_min - yPad` and `yStep` to a negative value so levels stack downward. (Historically `tickHeight` auto-computed as `yStep * 0.25` and went negative here, needing an explicit positive override — that's fixed: `tickHeight` now derives from `tickSize` and is always positive, so no override is needed.)

**`add_comparisons()` omnibus mode (v1.1):** `test` auto-classifies into pairwise (`mannwhitneyu`/`ttest_ind`/`ttest_rel`/`wilcoxon`/`tukey_hsd` — existing bracket behaviour) vs omnibus (`anova`/`kruskal`/`friedman`/`alexandergovern`). For omnibus: `_run_omnibus()` runs the test and the result is placed via a **single unified test label** (`testLabelPosition`, default `"auto"`). One label, content adapts: omnibus → the result (`ANOVA P = 0.003`); pairwise → the test **name** from `_TEST_DISPLAY` (`mannwhitneyu`→`Mann-Whitney U`, `wilcoxon`→`Wilcoxon signed-rank`, `tukey_hsd`→`Tukey HSD`, `dunn`→`Dunn's test`, `nemenyi`→`Nemenyi test`, `games_howell`→`Games-Howell`, etc.). `"auto"` resolves to `"topLeft"` for omnibus (shown) and `None` for pairwise (hidden, opt-in); an explicit preset draws it, `None` hides it. `testLabel` overrides the text; `testLabelX`/`testLabelY` (forwarded to `add_text`'s x/y) override the preset with manual coords; `testLabelOffsetX/Y` nudge. In omnibus+post-hoc the label shows the **omnibus result** (the post-hoc test name lives only in metadata). The old `omnibusPosition`/`omnibusLabel`/`omnibusOffsetX/Y` were renamed to `testLabel*` (v1.1, unreleased — no deprecation); `omnibusVerbose` stays (it tunes the omnibus content). The label block runs after `method` is resolved (so the pairwise name is available). If `pairs` is given, brackets are filled by the `postHoc` test (default per omnibus: anova→`tukey_hsd`, alexandergovern→`games_howell`, kruskal→`dunn`, friedman→`nemenyi`). `pairs` is now optional (was required-positional) — non-breaking. `omnibusVerbose=False` → terse `ANOVA P = 0.003`; `True` → `ANOVA F(2, 57) = 6.34, P = 0.003, η² = 0.18` (per-test statistic symbol/df/effect: F/df=(k-1,N-k)/η² for anova, H/(k-1)/ε² for kruskal, χ²/(k-1)/Kendall's W for friedman, A/(k-1)/η² for alexandergovern). Bracket p-value resolution is unified in `_bracket_pvalues()`: matrix post-hocs (`tukey_hsd`/`dunn`/`nemenyi`/`games_howell`) via `_post_hoc_matrix()` (corrected over all unique pairs; tukey ignores `correction`), pairwise tests per-pair + `_adjust()`. `nemenyi`/`friedman` require balanced data (raise `ValueError` otherwise). When no label is drawn and there are no `pairs`, returns an invisible `mark_point(opacity=0)` layer so it still composes with `+`.

**`add_comparisons()` report + metadata registry (v1.1):** Every `add_comparisons()` call builds a structured **record** via `statistics._make_record()` (the single source of truth: `kind`, a `dataChecksum` (see below), `test`, `groups` descriptives, an `omnibus` block, and a `comparisons` block carrying `test` (post-hoc/pairwise machine name), `correction` (`bonferroni`/`holm`/`None`; `None` for tukey since its correction is built-in, and when p-values are user-provided), and `pairs` with exact p-values + effect sizes as `{name, symbol, value}`) and **registers** it in the module-level `statistics._REPORTS` (`dict[str, dict]` keyed by content hash) via `_register_report()`, which returns a unique **marker name** `__dysonsphere_<hash>_<counter>`; `add_comparisons`/`add_correlation` tag their returned layer with `.properties(name=marker)`. `statistics._render_report(record)` turns a record into the plain text (title derived inside it: `Statistics | Omnibus | <name>` or `Statistics | Pairwise comparisons | <test|user p-values>`, box-drawing underline). **Report p-values are formatted by `_fmt_p()` at a fixed `_REPORT_SIGFIGS = 3` significant figures via `%g` (`= 0.032`, `= 1.22e-11`) — independent of the theme/on-plot `sigFigs` (so a cosmetic on-plot choice never coarsens the record); never floored to `P < 0.001` (that floor is only for the on-plot label via `_format_pvalue`). `_fmt()` (descriptives/effect sizes in the report) also uses 3 sig figs.** `_clamp_p()` maps a `0.0` p-value (scipy underflow — an impossible value) to `sys.float_info.min` at record-build, so neither the JSON nor the text ever reports `P = 0`; `_fmt_p` renders exactly that clamp value with `<` (`P < 2.23e-308`) since the true value is genuinely below float precision. `export.save()` drains records, renders text → `description`, AND embeds the raw records as JSON under `usermeta.dysonsphere.statistics` (see the usermeta design point) — so the data is machine-readable without parsing the text. **For an omnibus `test` the report covers ALL pairwise post-hoc comparisons** (`report_pairs` = every `i<j` pair, resolved via `_bracket_pvalues`/`pval_lookup`), even when only a subset is bracketed or `pairs=None`; for a pairwise `test` it covers exactly `pairs`. Brackets always use only `pairs` (looked up from `pval_lookup` by `frozenset`). The example `scripts/build/build_omnibus_example.py` → `docs/omnibus_example.png`; note `add_comparisons` brackets don't propagate the main chart's `sort`, so that example uses alphabetical-by-design category names to keep band order, and pads the y domain so the corner label clears the brackets. `report=True` also prints it; `save=True`/`save="dir"` writes `dysonsphere_report_<timestamp>.txt`. **Marker-based selection (v1.1, cures cross-chart contamination):** the registry is the only channel to carry a record from `add_comparisons` (a `+`-composed layer) to `save()`, because Altair strips custom `usermeta` across `+` — but the layer's Vega-Lite view **`name` survives `+`** (verified), so it's the tag. `save()` resolves the chart, walks the spec for `__dysonsphere_*` names (`metadata._scan_marker_hashes`), and embeds **only** the records whose markers are actually present (`statistics._select_reports`) — so a stats chart that's built but never saved can't contaminate a later `save()` of a different chart (the old "whatever's queued" drain was the bug). The queue is NOT cleared by `save()` (records persist so re-saving works); it accumulates, and public **`clear_stats()`** drops it (notebook housekeeping). Callable rebuilds re-register the same content hash (idempotent); the `_counter` keeps marker NAMES unique so a spec never has duplicate view names (which Vega compiles into `<name>_marks` datasets and rejects on collision → a render error). The marker is stripped from the written JSON (`metadata._strip_markers`), never appears in SVG/PNG at all, and is excluded (with `usermeta`) from the `vegaliteChecksum` (see the save() provenance point). Tests: `TestStatsQueueRobustness` in `tests/test_metadata.py`. Tests isolate the queue via an autouse fixture in `tests/conftest.py`.

**Record `dataChecksum` tagging (v1.1):** every `_make_record` / `_make_correlation_record` carries a **`dataChecksum`** field — the order-independent fingerprint of the dataframe the annotation was computed from, via **`utils.frame_checksum(df)`** (`ensure_polars(df).to_dicts()` → **`utils._hash_rows`**, the SAME multiset-of-per-row-digests core that backs the provenance `metadata._data_checksum`, extracted so there's one implementation). `add_comparisons`/`add_correlation` compute it (they already `ensure_polars(df)`) and pass `data_checksum=`; the builders default it to `None` (direct unit-test calls). Purpose: two annotations on **different** dataframes are distinguishable in `usermeta.dysonsphere.statistics` (you can't otherwise tell which record is which — the record only holds column names, not a group label), and identical-content frames match regardless of row order (the user's key requirement — survives `load()` + reorder). Because the field is part of the record, it also feeds the **content hash** (`_record_hash`, `sort_keys=True` so field ORDER is irrelevant), so two distinct-data annotations never collapse to one marker/record (the earlier failure mode). It is NOT rendered into the prose report (a sha256 isn't human-readable) — structured `statistics` only. **When the annotation is on the chart's inlined frame** (the common case) the record's `dataChecksum` **equals** `provenance.dataChecksum[i]`; it can differ only when the df isn't inlined (a subset used just for a fit line) or the inlined frame gained transform columns (`add_jitter`'s `jitter_x`) the raw df lacks. `_hash_rows` uses `default=str` so non-JSON cell types don't raise (JSON-native values unaffected → the provenance path stays byte-identical). Tests: `TestFrameChecksum` in `tests/test_utils.py`, `test_record_data_checksum_distinguishes_sources` / `test_comparisons_record_carries_data_checksum` in `tests/test_metadata.py`.

**`add_correlation()` (v1.1):** Sibling of `add_comparisons` for **continuous-x** scatters (not group comparisons). The method param is `method` (`"pearson"`/`"spearman"`/`"kendall"`, matching pandas `DataFrame.corr`) — NOT `kind` (`kind` is the record-type field, always `"correlation"` here). `method="pearson"` (default) computes `r`/`r²`/`P` + slope/intercept and draws the OLS line; `"spearman"`/`"kendall"` compute the rank coefficient (`ρ`/`τ`) + `P` and draw NO line (a straight line isn't their model — `line=` is a silent no-op). Pushes a `kind:"correlation"` record to the same `_REPORTS` registry (→ `usermeta.dysonsphere.statistics`). **Fit line:** a 2-point `mark_line` over the data's x-range from scipy's slope/intercept (single source of truth with the reported stats). Style is a 2-layer merge on top of the theme: the line **inherits the theme's `mark_line` config** by default (empty `mark_kwargs` → plain `mark_line()`; solid black `strokeWidth 0.375` under the built-in theme) → curated params (`color`/`strokeWidth`/`strokeDash`/`opacity`, each default `None` = inherit, set only when passed) → `lineStyle` dict passthrough (raw `mark_line` kwargs; **wins** collisions). `strokeDash` resolves `False`→`[0,0]`, `True`→theme `dashedWidth`, list→as-is. Does NOT reuse `_rule_mark_kwargs` (that forces `opacity`, preventing inheritance). `lineStyle` is the project's first raw-kwargs passthrough (no validation — bad keys error in Altair). **Axis gotcha (fixed 2026-07-07):** the fit-line sidecar's fields carry the REAL column names (`fit_df = {xCol: …, yCol: …}`, encoded non-shorthand via `alt.X(field=xCol, type="quantitative")` so `:`/`.` in names survive) with NO `title`/`axis` override. Why: Vega-Lite merges a shared axis's title by joining the layers' DISTINCT derived titles, so the old private `_x`/`_y` names CONCATENATED into the base chart's derived title (`"height, _x"`) whenever the base had no explicit `title=`; matching field names dedupe to one, and an explicit base title still beats this layer's derived one. `title=None` would null the base title and `axis=None` would suppress the axes entirely (both wrong). **Readout** (`_correlation_label`, on-plot via `add_text(position=...)`) is composed from independent parts: `coefficient` (`"r"` default / `"r2"` / `"both"` — Pearson only; rank always shows `ρ`/`τ`), `includePvalue` (default `False`), `includeEquation` (default `False`, Pearson only). So the DEFAULT readout is just `r = 0.85` (or `ρ = 0.81`). `verbose=True` is a shortcut that sets `coefficient="both", includePvalue=True, includeEquation=True` (overrides them) → `r = 0.85, r² = 0.72, P < 0.001, y = 0.84x + 0.27`. Negative intercepts render `- 0.27` (sign-aware). The readout uses `_format_pvalue` (floors to `P < 0.001`, display-only); the structured record/report always keeps the exact p. Example `scripts/build/build_correlation_example.py` → `docs/correlation_example.png`.

**Naming convention:** All public function parameters use camelCase (e.g. `xCol`, `yCol`, `bracketStyle`, `labelStyle`, `yPad`, `yStep`) to match Altair and Vega-Lite conventions. Private/internal functions (prefixed `_`) retain snake_case (e.g. `_pvalue_layer` uses `x_col`, `y_col`, `bracket_style` internally).

**`add_shade()` modes:** Two modes selected by which parameters are provided. **Square corners always:** every shade rect (all modes) pins `cornerRadius: 0` in its `mark_kwargs`, so `theme(cornerRadius=...)` — which styles `config.rect` and would otherwise round these background bands — never affects them. Shade rects are chart annotations, not data marks. Safety-net: `TestAddShadeCornerRadius` in `tests/test_annotations.py`.

*Band mode* (`categories` provided, `positions` omitted): shades every band on the x-axis. Color assignment: `palette[(i // repeat) % len(palette)]`. Consecutive same-color runs are merged into a single wider rect to eliminate the sub-pixel antialiasing seam that vl-convert PNG rasterization produces at coincident edges between adjacent same-color rects (src-over compositing partial coverage). Always operates on `axis='x'`.

*Positions mode* (`positions` provided): shades explicit coordinate ranges. Three sub-cases based on type:
- **String tuples** `(start, end)` — nominal axis, resolved to pixel coordinates via band scale formula so the shade layer doesn't participate in scale merging. Requires `categories`.
- **Numeric tuples** `(start, end)` — quantitative axis, positioned by **`alt.datum`** (`x`/`x2` or `y`/`y2`) which shares the main chart's scale WITHOUT contributing an axis title (a `title=None` field there clobbered the base title - fixed 2026-07-05).
- **Nested tuples** `((x_start, x_end), (y_start, y_end))` with `axis='both'` — draws intersection rects spanning both axes simultaneously. Each half is resolved independently using the same string vs. numeric logic, so mixed types (e.g. nominal x + quantitative y) work in the same rect. All rects render on a small `dummy_df` internal sidecar; string halves resolve to pixel `alt.value`, numeric halves to `alt.datum`.

**`add_shade()` parameters:**
- `palette` — list of hex colors to cycle through in light mode. Defaults to `colors["greys"]` when `None`. In dark mode this parameter is always ignored — `colors["greys"][-nShades:]` is used regardless. Resolved at call time from `alt.theme.options["darkmode"]` — callers must wrap chart construction in a callable passed to `ds.save()` for correct darkmode rendering.
- `nShades` — number of colors to use. Light mode: `palette[:nShades]`. Dark mode: `colors["greys"][-nShades:]`. Default `2`.
- `repeat` — number of consecutive ticks covered by each rect before advancing to the next color (band mode only). Default `1`.
- `opacity` — fill opacity. Default `1.0`.
- `stroke` — enable rect border. `False` (default) → no stroke. `True` → axis-style stroke: color is black/white (darkmode-aware), width is `axisWidth`.
- `strokeWidth` — explicit border width override in pixels. Only takes effect when `stroke=True`; overrides `axisWidth`. `None` (default) uses `axisWidth`.
- `strokeDash` — dash pattern for the rect border. `None` (default) → solid. `True` → inherit `dashedWidth` from the active theme. A list (e.g. `[4, 2]`) → use that pattern directly.
- `flush` — extend outermost rects to the axis domain edge. `None` (default) inherits from `theme(closed=...)`.
- `axis` — `'x'` (default), `'y'`, or `'both'`. Ignored in band mode (always `'x'`).

**`add_shade()` band scale formula:** band edges come from `utils.band_geometry(n, span)` (`scale="offset"`: band `i` spans `[starts[i], ends[i]]`). The offset variant is correct because `bandPaddingInner` only affects bar mark widths, NOT band boundary positions. With flush=False the outer gap on each side is `bandPadding * step`.

**`log_label_expr(base=10, notation="power")` in `nonlinear.py`:** Returns a Vega `labelExpr` string for typeset log-scale axis labels. Four notations: `"power"` (default) → `10⁴`, `2²⁰` (any base); `"scientific"` → `1×10⁴` (base=10 only, assumes exact-power ticks so mantissa is always 1); `"e"` → `1e+4` (base=10 only, uses `format(datum.value, '.0e')`); `"si"` → `10k`, `1M` (base=10 only, uses `format(datum.value, '~s')`). All notations except `"power"` raise `ValueError` for non-base-10. Power and scientific support exponents up to ±99 via `_SUP = "⁰¹²³⁴⁵⁶⁷⁸⁹"` — single-digit uses `_SUP[exp]`, two-digit uses `_SUP[floor(abs/10)] + _SUP[abs%10]`. Vega has no variable binding so `abs(round(log(datum.value) / log(base)))` is written out in full wherever needed. Callers pass the return value to `alt.Axis(labelExpr=...)` — independent of `add_log_ticks()`.

**`add_log_ticks()` design:** Lives in `nonlinear.py`. Wraps the caller's chart in `alt.layer(chart, minor_layer)` with `resolve_axis(y="independent")` or `resolve_axis(x="independent")`. The minor layer is an invisible `mark_point(opacity=0)` carrying a second axis with `ticks=True`, `labels=False`, `domain=False`, `grid=False`. Forcing `domain=[base**exp_min, base**exp_max]` on the minor layer's scale is critical — without it, Vega auto-fits the scale to the data extent, causing partial intervals at the edges with fewer visible minor ticks. For `base=10`: `minor_values = [m * 10**e for e in range(exp_min, exp_max) for m in range(2, 10)]`. For other bases: `minor_values = [base**(e + k/n_divs) for e in range(exp_min, exp_max) for k in range(1, n_divs)]` where `n_divs = nMinor + 1`. `_derive_exp()` uses `math.log10` for base 10 and `math.log(x, base)` otherwise. `axis='both'` uses `xField`/`yField` instead of `field` and resolves both axes independently in a single layer call. **`field` inference (single-axis mode):** when `field=None` and `axis` is `'x'`/`'y'`, it is inferred via `_infer_field(chart, axis)` — reads `chart.encoding.<axis>._kwds["shorthand"]` (the `.field` accessor returns a `_PropertySetter` descriptor, not the value — same Altair gotcha as `multilabel._strip_x_labels`), strips the `:Q` type suffix, and rejects aggregate/expression shorthands (`"(" in name`). Returns `None` (→ the caller raises a clear "pass field= explicitly" error) for a `LayerChart` (no top-level encoding), a data-less/complex channel, or an aggregate — i.e. inference fails **exactly** in the composed/multi-frame cases where `field` is ambiguous anyway, and always fails LOUD (raise, or a later `KeyError` on a mismatched df), never silently mis-inferring (the x/y-encoding field is definitionally the field the x/y minor ticks decorate). `df` stays explicit — it is also the numeric source for `_derive_exp`'s decade range, and making the 2nd positional optional would be a footgun. The inference path is covered against Altair-internal drift by `test_infers_field_from_{x,y}_encoding` in `tests/test_nonlinear.py` (they assert `_infer_field` against a real chart, so an Altair upgrade that breaks `_kwds` fails CI). `minorTickSize` defaults to `None` and is resolved at call time as `alt.theme.options.get("tickSize", 3) / 2` — so minor ticks automatically scale to half the active theme's tick length.

**`add_pow_ticks()` design:** Lives in `nonlinear.py`. Uses the same invisible-layer technique as `add_log_ticks()`. `majorValues` is required (cannot be auto-derived) and must match the `values=` passed to the main chart's `alt.Axis` — the minor layer uses them to infer interval boundaries and to set the independent scale domain via `domain=[min(majorValues), max(majorValues)]`. Minor tick `k` of `nMinor` between major ticks `a` and `b` is placed at `(a**exp + k/n_divs * (b**exp - a**exp))**(1/exp)` where `n_divs = nMinor + 1`. This interpolates linearly in power-transformed space, giving equal visual spacing. `axis='both'` requires `xField`, `yField`, `xMajorValues`, and `yMajorValues`. `minorTickSize` defaults to `None` and resolves at call time as `alt.theme.options.get("tickSize", 3) / 2`, matching the same convention as `add_log_ticks()`.

**`_fix_superscript_labels()` in export.py:** SVG post-processor that fixes misaligned Unicode superscript digits in scientific/power notation p-value labels (e.g. `10⁻¹⁴`). Root cause: Unicode superscript 1–3 (`¹ ² ³`, U+00B9/B2/B3, Latin-1 Supplement) and 0/4–9 (`⁰ ⁴–⁹`, U+2070/2074–2079, Superscripts block) live in different Unicode blocks with inconsistent vertical metrics in many fonts, causing visible misalignment when digits from both blocks appear in the same exponent. The fix: walk `<text>` and `<tspan>` elements via ElementTree (not raw regex, to avoid matching attribute values), find text matching `([×≈]\s*10)([⁰¹²³⁴⁵⁶⁷⁸⁹⁻]+)`, replace the exponent with a `<tspan dy="-2.5" font-size="4">` child element using plain ASCII digits and `−` (U+2212). Tuned for p-value label `fontSize=6` (exponent 4px, shift 2.5px). Called after `_simplify_svg` in `save()` and from the pvalue build script's manual pipeline. Must operate on the SVG file path (not the SVG string) so that attribute values — which Vega adds to marks as `aria-label` and `title` attributes containing the same label text — are never modified.

**`_italicize_stat_symbols()` in export.py (statistical-symbol italics):** SVG post-processor that typesets single-letter Latin statistical symbols in italic per scientific convention — Vega-Lite has no rich text (`fontStyle` styles a whole string), so each matched symbol is wrapped in `<tspan font-style="italic">` (the label font's true italic face; digits/operators stay upright). **Symbol set** (`_ITALIC_STAT_PATTERN`): `P` before `=`/`<`/`≈`; `F`/`H`/`A` before `(` (omnibus statistics); `W` before `=` (Kendall's W); `r` before optional `²` + `=` (the `²` digit stays OUTSIDE the tspan, upright); `n` before `=` (multilabel sample-size row); `y` before `=` and digit-preceded `x` before a `± b` term (fit equation); `t` before `-test`; `p`/`P` before `-value`/` value` (the p in "p-value" / "P-value" / "p value" - added 2026-07-12; `power` stays upright via the shared `(?<![A-Za-z])` guard); `U` after `Mann-Whitney `. **Greek stays upright by convention** (η², ε², χ², ρ, τ — deliberately absent from the pattern), and so does **`ns` (user decision 2026-07-07)**: it is an ABBREVIATION of "not significant", not a symbol — only single-letter symbols are italicized (the ISO/CSE rule; Prism/ggsignif render `ns` upright too; APA, which italicizes multi-letter abbreviations like `df`/`SD`/`ns`, is the outlier and is not followed). A whole-label italic `ns` shipped in 3.4.0-3.4.1 and was removed in 3.4.2. **GLOBAL policy (user decision 2026-07-07):** matches anywhere in rendered text — user `add_text("P = 0.03")` gets the identical treatment (same policy as `_fix_superscript_labels`; the typography is correct regardless of who wrote it, and hand-written labels stay consistent with generated ones). Each alternative is guarded (`(?<![A-Za-z])` etc.) so word-embedded letters (`power =`, `mean =`) never match. Processes `<text>` AND `<tspan>` targets (Vega sometimes wraps a label in an outer tspan); within an element only the string nodes it OWNS (`.text` + child tails) are rewritten, so each node is handled exactly once and existing children (e.g. superscript exponent tspans) are preserved in document order. **Must run AFTER `_fix_superscript_labels`** (that fixer only scans `.text`; italicizing first would hide the `×10⁻ⁿ` portion in a tail) — see the order in the export.py module entry. Like all fixers: `save()`/`show()` SVG+PNG only (HTML/bare-notebook render upright), never touches `aria-label`/`title` attributes. **Font prerequisite — the theme default font is the FAMILY name `"Helvetica Neue"`, not the PostScript name `"HelveticaNeue"` (changed 2026-07-07):** vl-convert's rasterizer matches the PostScript name to the regular face only, so no italic variant is ever found (and it does not synthesize oblique) — italic tspans silently rendered upright in PNGs. Both names produce byte-identical SVG output otherwise (verified: identical metrics/positions), so the rename is rendering-inert except for unlocking the italic face. **The website ports this fixer client-side** (`italicizeStatSymbols` / the `ITALIC_STAT` regex in `website/src/lib/fixSuperscripts.ts`, "mirrors `export._ITALIC_STAT_PATTERN` exactly") so live vega-embed charts match exports - **keep the two patterns in sync** (the p-value rule was added to both together). Tests: `TestItalicizeStatSymbols` in `tests/test_export.py`.

**Grid annotation alignment gotchas:** `_multilabel_layer()` suppresses the y-axis and renders row labels as explicit `mark_text` marks (not axis labels) because Vega-Lite's axis rendering pipeline doesn't guarantee pixel-perfect alignment with `mark_text` even when both use `baseline="middle"`. `bandPosition=0.5` is set explicitly on the shared `y_enc` because per-mark defaults vary across mark types. `align="center"` is required on all `mark_text` content marks — without it, vertical band placement drifts in some Vega-Lite versions. The `strokeDash=[0, 0]` on the connecting rule overrides the theme's `dashedRule=True` default. The `orientation` parameter (`"vertical"` default, `"horizontal"` for legacy behavior) controls the direction of connecting lines. **Vertical:** uses `mark_line` + `detail="__line_id:N"` (two rows per segment in `line_rows`, one per endpoint) at fixed `x` — NOT `mark_rule` + `y2`, because `y2` with a different field name pollutes the shared ordinal y scale and corrupts `paddingInner`, shifting row spacing. **Horizontal:** uses `mark_rule` + `x`/`x2` encoding at fixed `y`. **`domain=row_order`** is set explicitly on the y scale (`alt.Scale(domain=row_order)`) to prevent Vega-Lite's shared-scale domain union from reordering rows when connecting lines are present. **Mixed styles:** `rowStyles: dict[str, str] | list[str] | None` overrides `style` per row; accepts a dict mapping row labels to styles, or a list of styles in `row_order`; non-bool values always force `"text"` per row regardless. Style branches (plusminus, text, symbol) are now separate `alt.layer()` entries rather than early-return branches. Connecting lines are computed only over symbol-style rows; non-symbol rows between symbol rows are skipped in run detection (no error). **`rowHeight`** defaults to `10` px (flat). `_strip_x_labels` in `add_multilabel` reads `x._kwds` directly because `.axis` on `alt.X` returns a `_PropertySetter` descriptor, not the stored value. **Tick-to-annotation alignment:** the annotation columns are placed by the band formula (`step = chartWidth / (n + 2*bandPadding)`), and axis ticks land on the exact band centres at render time (`tickRound: false` / `axisBand.tickOffset: 0` in the theme config - see the exact-tick-positions design point), so the two coincide with no post-processing. **Grid-to-content alignment fix:** on an open plot each axis is drawn `axisOffset` px away from the plot (`axisOffset` defaults to `tickSize * 1.5` - resolved once in `theme()`, see the offset-defaults point - or `0` when `closed=True`): the x-axis group at `y = chartHeight + axisOffset`, the y-axis group at `x = -axisOffset`. Vega renders each grid line INSIDE its axis group, so every grid line inherits that offset and renders dragged toward its axis: the vertical (x-axis) grid lines shift DOWN (`ty=-chartHeight`/`y2=chartHeight` within a group at `+axisOffset` → top `axisOffset` short of the highest tick, bottom overshooting onto the detached x-axis), and the horizontal (y-axis) grid lines shift LEFT (`translate(0,y)`/`x2=chartWidth` within a group at `-axisOffset` → left end on the detached y-axis, right end `axisOffset` short of the content). `export._align_grid_to_content()` translates each line back by `axisOffset` (**span unchanged**): vertical lines UP (`ty -= axisOffset`), horizontal lines RIGHT (`tx += axisOffset`) - distinguished by whether `|y2|>|x2|`. So BOTH grid directions span the plot content exactly and float symmetrically off both detached axes (this symmetry was the fix: an earlier version lifted only the vertical grids, leaving the horizontal grids touching the y-axis - visibly asymmetric). `_render_fixed_svg` computes the resolved offset (`0 if closed else axisOffset`) and skips the call at `0`, so closed plots (axes already flush, grid already spanning the content) are untouched. Any script that builds a grid-labels chart must render via `ds.save()` / `export._render_fixed_svg()` (not a bare `chart.save()`) to get the grid alignment and the other SVG post-processors; pass a callable to `ds.save()` when dot colours depend on darkmode.

**Grid annotation darkmode:** Dot colours in `_multilabel_layer(style="symbol")` are resolved from `alt.theme.options` at call time. In dark mode: `positive_color="white"`, `negative_fill=colors["greys"][11]`, `negative_stroke="white"`. Pass a callable to `ds.save()` so the chart rebuilds after each darkmode toggle: `ds.save(lambda: ds.add_multilabel(..., style="symbol"), "plot")`.

**`add_multilabel` spans:** `span: dict[str | None, list[str]] | list[dict[str | None, list[str]]] | None` groups contiguous x-axis categories under a shared rule or bracket. Keys are span labels (`""` or `None` for no label); values are lists of categories. Use a list of single-entry dicts when multiple unlabeled spans are needed (plain dict collapses duplicate `None` keys). The span extends from `min(index)` to `max(index)` of the listed categories. All marks use `alt.value()` for pixel coordinates — no `:Q` scale is added to the layer, avoiding scale merging conflicts with the annotation's `:N` x scale. `spanBracketStyle="line"` (default) draws a plain horizontal rule; `"bracket"` adds vertical end ticks. `spanLabelPosition="bottom"` (default) places the label below the rule; `"top"` places it above. `spanBracketReverse=True` (default) points ticks toward the annotation rows (no height added); `False` extends ticks below the rule (adds height). `spanTickHeight` defaults to `alt.theme.options.get("tickSize", 3)`. `spanGap` defaults to `rowHeight × 0.3`. X positions: `step = chartWidth / (n + 2 * bandPadding)`, `x1 = step * (bp + 0.5 + i_min) - step * 0.30`, `x2 = step * (bp + 0.5 + i_max) + step * 0.30` — centered on band centers with ±0.30-step extension. Rule and tick colors are darkmode-aware; `strokeDash=[0, 0]` overrides `dashedRule`. **`defer_cat_label`:** when `categoryLabel=True`, `categoryLabelPosition="bottom"`, and `span` is present, the category label row is deferred to below the spans so the visual order is rows → spans → category labels.

**`add_multilabel` sample size row:** `showSampleSize=True` injects a per-category count row using `count_n(df, xCol, categories)` from `utils.py`. The row is always forced to `style="text"` via an explicit `rowStyles` override — belt-and-suspenders on top of `_row_style`'s non-bool auto-detection. When `rowStyles` is a list, it is converted to a dict using the original `groups` keys *before* the n-row is inserted, so list indices remain correct. `sampleSizeIndex` sets insertion position using `list.insert()` semantics (default `0` = first; `len(groups)` = last; `-1` = second-to-last). `sampleSizeLabel` sets the row label (default `"n ="`). `groups` defaults to `{}` and `categories` defaults to `None`, so calling `add_multilabel(chart, categories=CATEGORIES, showSampleSize=True, df=df, xCol="group")` is valid with no explicit `groups`. These parameters are on `add_multilabel` only, not `_multilabel_layer`.

**`add_multilabel` category labels:** `categoryLabel=True` renders x-axis category names as angled text in a dedicated row via a separate `mark_text` layer using `alt.value(label_y)` absolute positioning (not part of the band scale). `categoryLabelPosition="top"` or `"bottom"` (default) places the row above or below all data rows — intermediate positions are not supported. `categoryLabelAngle` (default `-45`) controls rotation; Vega-Lite requires angles in [0, 360], so the value is passed as `categoryLabelAngle % 360`. `align` is `"center"` when `categoryLabelAngle % 360 == 0`, `"right"` otherwise. `baseline="middle"` is used universally (not `"top"`) so that the text is correctly centered for all angles including `−90°`. The anchor y position is `label_y_offset = fontSize/2 × |cos(angle)|` for the top case (shifts anchor down to prevent upward clipping), and `n*rowHeight + label_y_offset + extra` for the bottom case where `extra = max(0, categoryLabelHeight − tight_height)` shifts the anchor further into the label row when `categoryLabelHeight` is larger than the tight-fit value. `categoryLabelHeight` auto-computes as `ceil(fontSize × 0.6 × max_len × |sin(angle)| + fontSize × |cos(angle)|)` — the rotated bounding box of the longest label. The band scale is shifted via `alt.Scale(range=[categoryLabelHeight, n*rowHeight + categoryLabelHeight])` for `"top"` so data rows start below the label row. The category label row lives outside the band scale; nothing (including the n= row) can be placed between data rows and category labels at the far end.

**Grid annotation hconcat label overflow:** Row labels are positioned outside the declared `width` via `alt.value(x)` and are not clipped or auto-spaced by Vega-Lite's layout engine. In `hconcat`, labels from one panel can bleed into adjacent panels; add explicit `spacing` or outer padding to compensate.
